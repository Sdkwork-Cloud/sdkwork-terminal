# 产品设计与需求范围

## 1. 文档定位

本文档定义 `sdkwork-terminal` 的产品定位、目标用户、能力边界、优先级与非目标，用于冻结 PRD 级需求范围。

## 2. 当前事实

- 当前产品是 `Terminal 应用`，不是代码编辑器，也不是聊天聚合器。
- 当前产品主参考对象是 `Windows Terminal`，辅以 `VS Code Terminal`、`WezTerm`、`Zellij`、`iTerm2`、`Ghostty`。
- 当前集成对象是 `Codex`、`Claude Code`、`Gemini`、`OpenCode` 等原生 CLI，而非模型 API 抽象层。
- 当前需要同时覆盖 `桌面单机` 与 `Server Mode 自托管` 两类产品形态。
- 当前官方支持边界必须写清楚：桌面基线为 `Windows / Ubuntu / macOS`，服务端基线为 `Ubuntu Server`，CPU 基线为 `x64 / arm64`。

## 3. 一句话定位

一款对标行业专业标准、以 `Terminal / Session / Runtime` 为核心、原生支持 Shell/SSH/Docker/Kubernetes/AI CLI、并可从桌面版平滑演进到 Server Mode 的跨平台 Terminal 平台。

## 4. 目标用户

- 高频使用 Terminal 的开发者。
- DevOps、SRE、平台工程师、云原生工程师。
- 同时使用本地 Shell、SSH、Docker、Kubernetes 的高级用户。
- 高频使用 `Codex`、`Claude Code`、`Gemini`、`OpenCode` 等 AI CLI 的用户。
- 需要自托管终端工作台、远程 Session 与审计能力的团队。

## 5. 核心场景

### 5.1 本地开发工作台

- 多窗口、多标签、多 pane 并行工作。
- 按项目或工作区组织 Session、Profile、快捷命令和 AI CLI 模板。
- 在同一工作台同时运行 Shell、构建、测试、TUI 工具和 AI CLI。

### 5.2 远程与基础设施运维

- 连接 SSH 主机并长期保持 Session。
- 进入 Docker 容器与 Kubernetes Pod。
- 在远程目标中启动或托管 AI CLI Session。
- 对长任务、弱网、断线重连、Detach/Attach 有稳定支持。

### 5.3 AI CLI 原生工作流

- 自动发现本机或远程环境中的 CLI 二进制。
- 以 Profile/Template 方式启动不同 CLI。
- 将 CLI Session 纳入统一的 Session Center、历史、恢复、回放与诊断体系。

### 5.4 Server Mode 团队交付

- 多用户、多工作区、多 Runtime Node 管理。
- 通过桌面端或 Web Console 接入同一控制面。
- 支持 Docker、Kubernetes 部署与企业治理能力扩展。

## 6. P0 需求范围

### 6.1 Terminal 基础能力

- 多窗口、多标签、多 pane。
- Profile、Theme、Keybinding、Command Palette。
- Search、Scrollback、Selection、Clipboard、Hyperlink。
- Unicode/CJK/IME、Bracketed Paste、Alt Screen、Mouse Reporting。

### 6.2 Session 能力

- Create / Attach / Detach / Resume / Replay / Terminate。
- Session Center、历史记录、会话标签、会话恢复。
- 本地退出 UI 后 Session 仍可持续存在。

### 6.3 运行目标能力

- Local Shell。
- SSH。
- Docker Exec。
- Kubernetes Exec。
- Remote Runtime。
- Server Mode Runtime Node。

### 6.4 AI CLI 原生集成

- `Codex`
- `Claude Code`
- `Gemini`
- `OpenCode`

统一要求：

- 原生命令、原生参数、原生认证、原生交互、原生输出。
- 平台只负责发现、启动、托管、恢复、观测。

### 6.5 Server Mode 基础能力

- 用户、工作区、Runtime Node、Session Broker。
- 审计、策略、资源注册、Web Console。
- Docker / Kubernetes 标准化部署。

### 6.6 平台与交付基线

- Desktop `P0`：`Windows x64`、`Ubuntu x64`、`macOS arm64`。
- Desktop `P1`：`Windows arm64`、`Ubuntu arm64`、`macOS x64`。
- Server `P0`：`Ubuntu Server x64 / arm64`。
- 发布物必须按目标三元组独立构建、独立测试、独立发布，不允许用“理论跨平台”替代实际证据。

## 7. P1-P2 需求范围

### 7.1 P1

- Session 录制与命令级回放。
- 高级 Shell Integration。
- 团队模板与共享配置。
- 企业安装器、自动升级与回滚标准化。

### 7.2 P2

- SSO、细粒度 RBAC、合规审计。
- 策略治理、配额治理、租户隔离。
- 扩展市场与受控插件体系。

## 8. 非目标

- 不做通用 IDE。
- 不内置完整代码编辑器替代方案。
- 不把 AI CLI 抽象成统一 Provider 协议。
- 不在第一阶段追求所有私有终端扩展协议的 100% 覆盖。
- 不把产品做成“聊天窗口外加一个 Terminal 面板”。

## 9. 成功标准

- 专业用户愿意把它作为主力 Terminal。
- 本地、远程、容器、Kubernetes、AI CLI 场景都顺手且稳定。
- 桌面版与 Server Mode 共用同一核心模型，而不是两套产品。
- 架构可长期演进，不因功能增长而退化为巨型单体。
- 平台支持声明与证据一致：没有目标三元组、安装、启动与核心交互证据，就不能宣称已支持该平台/架构。

## 10. 评估标准

| 评估项 | 合格线 | 领先线 | 当前判断 |
| --- | --- | --- | --- |
| 产品定位清晰度 | 明确是 Terminal 平台 | 与 IDE/AI 抽象平台边界完全清晰 | `L4` |
| 需求完整度 | 覆盖桌面、远程、AI CLI、Server Mode | 覆盖实施、交付、治理、商业化 | `L4` |
| 范围控制力 | 有明确非目标 | 能防止需求膨胀和语义漂移 | `L4` |
