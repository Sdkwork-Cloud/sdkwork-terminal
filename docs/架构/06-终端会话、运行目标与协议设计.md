# 06-终端会话、运行目标与协议设计

## 1. 目标

冻结 `Session` 一等实体模型、`Attach / Detach / Reattach / Replay / Recovery` 语义、`ExecutionTarget` 边界，以及 Desktop 与 Server 共享的会话治理原则。

## 2. 当前事实

- `Step 06` 已落地 `session-runtime` 与 `replay-store` 的基础主链。
- 当前 Session 真相已从 UI 下沉到 runtime，前端只消费 Session/Attachment 派生视图。
- Desktop `SessionRuntime` 的 SQLite bootstrap 已收口到共享 Rust `sdkwork-terminal-control-plane`，`src-tauri` 只负责解析 `app_local_data_dir()` 并接入 IPC，不再自带平行持久化初始化分支。
- `Step 07` 已冻结 `ExecutionTargetDescriptor`、connector health、transport、launch/probe request 字段，并将 launch/probe/read 读链接入桌面端。
- connector-backed Session 已支持 `launch_intent` 持久化、`Starting -> Running / Failed` 显式收口、replay 证据写入。
- `sdkwork-terminal-pty-runtime` 已支持显式命令 PTY bootstrap，`sdkwork-terminal-control-plane` 已支持共享 `connect -> admit -> attach -> spawn -> resolve` 编排，桌面端已新增 `desktop_connector_session_create` 作为 interactive connector create 入口。
- `packages/sdkwork-terminal-shell` 已新增 tab 级 `runtimeBootstrap`，ShellApp 现可按 tab 事实在 `createLocalShellSession(...)` 与 `createConnectorInteractiveSession(...)` 间分支，connector interactive session 已可承接为顶部 live terminal tab。
- 顶部 `Connectors` 正式产品入口现已通过 desktop bridge 读模型接入 ShellApp；`desktop_execution_target_catalog` 不再返回静态演示目录，而是消费共享 Rust `connector discovery` 结果并在桌面端按需刷新。
- connector launch intent 的默认 `requestId` 必须是 action-scoped，而不是 target-scoped；同一 authority 的重复打开必须每次生成新的请求标识，否则 ShellApp 的去重保护会把第二次 live tab 打开误判为重复动作。
- 当前 discovery baseline 已冻结为：`SSH=env 优先 + ~/.ssh/config 多 Host`、`Docker=env 优先 + docker ps 多运行容器`、`Kubernetes=env 优先 + current-context + 多 pod`、`Remote Runtime=env authority only`；health/readiness 与 `sessionLaunchable` 由真实 host/config/health 诊断决定。
- `tools/smoke/connector-interactive-probe.mjs` 已作为 Step 07 的仓库级证据入口落地，用于固化 `ssh / docker-exec / kubernetes-exec` live terminal、Session Center reattach 与 restart/recovery 的 smoke 计划、report template 与 review template。
- `tools/smoke/connector-interactive-probe.mjs --print-preflight` 已把 Step 07 主机 readiness 审计仓库化，允许在进入实机 smoke 前先显式识别 `docker daemon`、`kubectl current-context`、authority 发现等环境阻塞。
- `tools/smoke/connector-interactive-probe.mjs --print-batch-plan` 已把三平台三目标的实机证据采集矩阵仓库化，明确 `platform-target` smoke 可并行、`docs/step + docs/release` 回写必须串行。
- `tools/smoke/connector-interactive-probe.mjs --write-batch-templates` 已把 Step 07 connector report/review skeleton 的文件化输出仓库化，允许按平台批量预生成模板后再做实机执行。
- `tools/smoke/connector-interactive-probe.mjs --print-execution-plan --platform <platform>` 已把“当前主机 preflight 事实 + 当前平台 batch 矩阵”收口成执行视图，显式给出 `executableNow / skipReason / blocked/tool-only`，避免把 host 阻塞混入产品回归判断。
- `tools/smoke/connector-interactive-probe.mjs --write-batch-templates --platform <platform> --output-dir <dir> --ready-only` 已支持按当前主机 readiness 过滤模板输出，只把 ready targets 投入本轮 field smoke，blocked entries 保留在 skipped 清单中等待环境清障。
- 当前剩余缺口收敛为 `ssh / docker-exec / kubernetes-exec` 的跨平台 reattach/recovery 实机证据、多目标 inventory 深化与 Remote Runtime 主链；当前仍不能宣称所有 connector 已达到完整交互式终端能力。

## 3. Session 一等实体模型

### 3.1 核心原则

- Session 拥有稳定 `sessionId`。
- Session 独立于窗口、标签页、WebView、浏览器 tab 存在。
- 宿主 UI 卸载、面板切换、工作台重建都不能等价于 Session 丢失；显式 `Close tab` 若被产品定义为结束任务，则属于 `Terminate` 而非 `Detach`。

### 3.2 Attachment 语义

- Attachment 拥有稳定 `attachmentId`。
- Attachment 只是观察与写入句柄，不转移 Session 所有权。
- Attachment 可以断开、重建、重新附着。

### 3.3 平台归属字段

每个 Session 至少应显式可追踪：

- `sessionId`
- `attachmentId`
- `targetType`
- `hostMode`
- `platformFamily`
- `cpuArch`
- `runtimeLocation`
- `launchIntent`
- `lastAckSequence`
- `replayCursor`

说明：

- `platformFamily` 只允许声明真实目标，例如 `windows`、`ubuntu-desktop`、`macos`、`ubuntu-server`。
- 不能使用模糊的“跨平台”或“linux-like”作为权威平台标识。

## 4. 生命周期语义

| 状态 | 含义 |
| --- | --- |
| `Creating` | 会话实体已创建，资源尚未就绪 |
| `Starting` | 正在建立底层运行时 |
| `Running` | 可持续交互 |
| `Detached` | Session 仍存在，但当前无活跃 attachment |
| `Reattaching` | 正在恢复 attachment |
| `Replaying` | 正在读取历史片段 |
| `Stopping` | 正在退出 |
| `Exited` | 已退出，保留历史与元数据 |
| `Failed` | 启动或运行失败，需要诊断 |

规则：

- 所有 `ExecutionTarget` 都必须共享同一状态机。
- connector 或 CLI 只能扩展接入方式，不能扩展第二套状态机。
- `Starting` 只能由明确的 launch resolution 或等价权威事件收口。

## 5. Reattach / Replay / Recovery 边界

### 5.1 Reattach

- 面向活跃 Session。
- 目标是恢复实时交互。
- 必须依赖 Session 真相源，而不是旧 UI 缓存。

### 5.2 Replay

- 面向历史片段读取。
- 目标是读取 transcript、markers、state/output/warning/exit 证据。
- 不能被误判为“已恢复实时交互”。

### 5.3 Recovery

- 面向崩溃、断连、重启、弱网等场景。
- 优先依赖 `launchIntent`、持久化索引、ack、cursor 与目标环境诊断信息。
- Desktop 与 Server 必须共享恢复语义，但恢复实现可因宿主差异不同。

## 6. Ack / Cursor / Replay Evidence

- `lastAckSequence`：attachment 已确认位置。
- `cursor`：replay 读取游标。
- `replayStateAck`：只读表达 attachment ack 相对 latest state 的位置。
- `replayStateOutputDelta`：只读表达 latest state 与 latest output 的相对先后。
- `replayEvidenceAck`：只读表达 attachment ack 相对 latest warning/exit evidence 的位置。
- `replayEvidenceFreshness`：只读表达 latest warning/exit evidence 相对 `observedAt` 的时效性。

规则：

- 上述摘要都属于“读侧可见性”，不代表 interactive attach、实时 streaming 或 replay cursor control 已完成。

## 7. ExecutionTarget 边界

统一运行目标冻结为：

- `local-shell`
- `ssh`
- `docker-exec`
- `kubernetes-exec`
- `remote-runtime`
- `server-runtime-node`
- `cli-native`

### 7.1 local-shell

- Windows：基于 `ConPTY`
- Ubuntu/macOS：基于 `POSIX PTY`

### 7.2 remote targets

- `ssh / docker-exec / kubernetes-exec / remote-runtime / server-runtime-node` 统一共享 Session 生命周期。
- 区别只能体现在连接方式、启动方式、诊断方式与部署位置。

## 8. 协议设计规则

- `ConnectorSessionLaunchRequest` 是 connector-backed Session 的统一 admission 入口。
- `SessionCreateRequest::from_connector_launch_request` 负责把 launch request 映射到 runtime admission。
- Desktop 组合层可以桥接 launch/probe/read，但不能把 bounded read 伪装成实时 attach。
- `desktop_connector_launch`、`desktop_connector_exec_probe`、`desktop_session_index`、`desktop_session_replay_slice` 都是结构化能力入口，不是完整实时流式协议的替代品。
- `desktop_connector_session_create` 是 desktop host 的 interactive connector admission 入口；其 accepted truth loop 必须是
  `connect preflight -> Session admission -> attachment create -> PTY spawn -> launch resolution -> runtime replay/event`，
  不能退化成只返回“connect 成功”的假会话。
- 顶栏 connector product intent 必须遵守“同一动作、同一 `requestId`；重复动作、全新 `requestId`”规则；禁止把 `requestId` 默认绑定为 `targetId + authority` 这类稳定键，否则会破坏多 tab 并发打开语义。
- Desktop 宿主上的 Session 持久化 bootstrap 必须优先复用共享 Rust 控制面能力；Tauri host 只允许补充路径解析、事件发射与 IPC 桥接，不允许复制第二套 SQLite SessionRuntime 初始化逻辑。
- `desktop_session_attach`、`desktop_session_detach`、`desktop_session_reattach` 是 desktop host 对 `SessionRuntime` 的结构化 Session 写入口；前端不得跳过 bridge 直接假造 attachment 状态。
- Desktop 本地 shell attachment 必须存在显式 ack 回写链；消费 replay/event 后，桌面端必须通过 `desktop_session_attachment_acknowledge` 回写 `attachment.lastAckSequence / cursor`，不能让 Session Center 永久读取到陈旧 attachment 状态。
- Host teardown 期间必须 best-effort `detach` 仍存活 attachment；`Detached` Session 后续即使继续收到后台 output，也必须保持 `Detached`，直到新的 attachment 真正建立。
- Session Center 在产品层只能以 terminal chrome 触发的 `overlay / drawer / palette` 形态进入，不能回退成 dashboard/home 页面，否则会破坏 `terminal-first` 首页约束。
- 当前 desktop 交互式 `reattach` 对 `target=local-shell / ssh / docker-exec / kubernetes-exec` 提供正式保证；`remote-runtime` 在其 live attach/input/stream host 未闭环前，只能在 Session Center 中作为读侧会话真相展示，不得伪装成可输入终端。

## 10. 评估标准

| 项目 | 标准 | 当前要求 |
| --- | --- | --- |
| Session 一等实体 | Session 独立于 UI 生命周期 | `L4` |
| 生命周期一致性 | 所有 target 共享状态机 | `L4` |
| Reattach / Replay / Recovery | 语义明确、边界清楚 | `L4` |
| 平台显式性 | `Windows / Ubuntu / macOS / Ubuntu Server` 可追踪 | `L4` |

## 11. 结论

会话设计的关键不是“把终端输出存起来”，而是建立一套跨宿主、跨目标、跨平台都成立的 Session 真相模型，并明确哪些是实时交互、哪些只是结构化读侧能力。

## 2026-04-10 Supplement - Desktop SQLite Recovery Bootstrap

- Desktop Session recovery 的共享入口已经冻结为 `sdkwork-terminal-control-plane::create_desktop_session_runtime(...)`。
- Tauri host 解析到 `app_local_data_dir()` 后，必须落到 `session-runtime.sqlite3`，并交给共享控制面初始化；这保证桌面端与后续 server-side control plane 复用同一条持久化 bootstrap 语义。
- 当前 Windows 开发主机上的 `src-tauri` Rust 单测二进制仍存在 `STATUS_ENTRYPOINT_NOT_FOUND` 原生加载限制；因此桌面恢复主链的自动化证据优先来自共享 Rust 控制面测试，`src-tauri` 本地只要求 `cargo check` 通过并保持桥接接线正确。

## 2026-04-10 Supplement - Runtime-Node Server Recovery Bootstrap

- `sdkwork-terminal-runtime-node` 现已提供 `create_runtime_node_session_runtime(...)` 与 `RuntimeNodeBootstrapConfig`，用于冻结 server-side `SessionRuntime` bootstrap。
- 该 bootstrap 必须显式产出 `hostMode=server`、`platformFamily`、`cpuArch`、`runtimeLocation` 与 `storageSurface`，禁止把 `Ubuntu Server` 恢复证据继续写成模糊的“server persistence”。
- `server-runtime-node` 的 Step 06 恢复语义已经收口为：persisted `session index / replay / lastAckSequence` 能在 runtime-node 进程重建后恢复；这属于 Session truth loop，不等价于 Step 10 broker/runtime-node 协调闭环。

## 2026-04-10 Supplement - Remote Runtime Web Bridge

- `remote-runtime` 的 accepted create path 已收口为：
  `Resource target -> createRemoteRuntimeSessionCreateRequest -> WebRuntimeBridgeClient.createRemoteRuntimeSession -> publicApi /sessions`。
- `remote-runtime` 的 accepted stream/read path 已收口为：
  `publicApi /replays + runtimeStream /attach`；
  不允许复用 desktop `system-cli` connector request、Tauri IPC、或本地 PTY spawn 来伪造远端 runtime attach。
- `createWebRuntimeBridgeClient(...)` 现在提供 `sessionIndex / createRemoteRuntimeSession / sessionReplay / writeSessionInput / writeSessionInputBytes / resizeSession / terminateSession / subscribeSessionEvents` 基线，供 `apps/web` 与后续 runtime-node host 接线。
- desktop 边界保持不变：`remote-runtime` 仍只允许出现在 discovery/read-model/Session Center truth 中；在真实 host attach/input/stream/recovery 闭环完成前，不开放 desktop interactive reattach。

## 2026-04-10 Supplement - ShellApp Connector Bootstrap

- `packages/sdkwork-terminal-shell` 现已引入 tab 级 `runtimeBootstrap`，desktop Shell tab 不再被强制视为 `local-shell`。
- ShellApp 当前 accepted truth loop 为：
  `open tab -> resolve tab runtimeBootstrap -> createLocalShellSession/createConnectorInteractiveSession -> bind runtime session -> replay/event -> live input/resize/terminate`。
- `src-tauri::desktop_execution_target_catalog`、desktop bridge `executionTargets()` 与 `apps/desktop::loadDesktopResourceCenterSnapshot(...)` 现已把 desktop execution target catalog 接入顶栏 `Connectors` 产品入口；ShellApp 不再依赖模块加载时的静态 connector 常量。
- desktop profile/connectors dropdown 每次打开前都会触发 `resourceCenterSnapshot` 刷新，desktop host 现在具备“桥接 fresh discovery -> snapshot 投影 -> header 产品入口”的真实闭环。
- `apps/desktop/src/connector-shell.ts` 现已为默认 connector launch intent 生成一次性 `requestId`，修复“同一 target 从顶栏重复打开时被 ShellApp 去重吞掉”的产品缺陷。
- desktop bridge / ShellApp / Tauri IPC 现已统一收口到 `session-first` 写链路：`writeSessionInput / writeSessionInputBytes / resizeSession / terminateSession / subscribeSessionEvents` 对应 `desktop_session_input / desktop_session_input_bytes / desktop_session_resize / desktop_session_terminate`。
- `crates/sdkwork-terminal-control-plane` 当前已对 `ssh / docker-exec / kubernetes-exec` 共享 interactive bootstrap truth loop 建立自动化覆盖；下一阶段重点已转为三类 connector 的实机 smoke 归档与 reattach/recovery 闭环，而不是重新发明第二套 admission 语义。
- connector interactive session 现在已经能承接为顶部 live terminal tab；剩余缺口在于 `ssh / docker-exec / kubernetes-exec` 的实机恢复证据、多目标 inventory 深化与 Remote Runtime recovery。
- desktop 输入热路径现在允许 `idle / binding / retrying` tab 进入 pending queue，避免 connector/local shell 在 bootstrap 早期直接丢键。
## 2026-04-10 Supplement - Web Shell Runtime Binding

- `apps/web` 的 accepted Step 07 产品链路现已收口为：
  `VITE_TERMINAL_RUNTIME_* -> createWebRuntimeBridgeClient -> ShellApp(webRuntimeClient, webRuntimeTarget) -> tab.runtimeBootstrap(remote-runtime) -> createRemoteRuntimeSession -> publicApi /replays -> runtimeStream /attach -> write/resize/terminate`。
- `remote-runtime` 已成为 `ShellApp` 的 tab 级一等 bootstrap 语义，和 `local-shell / connector` 并列；其 request 真相必须保存在 tab model 中，支持 restart/duplicate 后继续解析，而不是依赖 React 临时状态拼接。
- `showLivePrompt` 现在只允许出现在“web 未配置 runtime target”的显式回退场景；一旦 tab 走 `remote-runtime`，即使仍处于 `binding / retrying`，也必须保持 terminal stage + runtime status overlay，而不是退回伪输入框。
- desktop 与 web 共享 `sessionReplay / writeSessionInput / writeSessionInputBytes / resizeSession / terminateSession / subscribeSessionEvents` 热路径；`detachSessionAttachment / acknowledgeSessionAttachment` 继续严格限制在 desktop host。
- 本轮仍不改变 desktop 边界：`remote-runtime` 仅新增 web shell binding，不新增 desktop interactive reattach 保证。

## 2026-04-10 Supplement - Runtime-Node Host Core

- `crates/sdkwork-terminal-runtime-node/src/host.rs` 现已提供纯 Rust `RuntimeNodeHost`，accepted host loop 为：  
  `RemoteRuntimeSessionCreateRequest -> SessionRuntime.create+attach -> LocalShellSessionRuntime.create_process_session -> replay/state -> input/input-bytes/resize/terminate -> output/warning/exit -> sqlite recovery`。
- `RuntimeNodeHost` 当前只承载 Step 07 所需的 server-side session truth 与 live PTY orchestration；`publicApi / runtimeStream`、broker、调度与多节点协同继续留在后续薄包装/Step 10，不允许把第二套 Session 模型重新塞回 host。
- 该 host 现已显式暴露 `sessionIndex / sessionReplay / writeSessionInput / writeSessionInputBytes / resizeSession / terminateSession / subscribeSessionEvents`，与 web bridge 的 session-first 热路径逐项对齐。
- `crates/sdkwork-terminal-pty-runtime::terminate_session()` 现已先摘除 runtime handle，再执行 best-effort close；Windows `os error 6 / 1460` 不再阻塞 `Exit` 收口，从而避免 runtime-node session 长期停留在 `Stopping`。
- desktop 边界仍保持不变：`remote-runtime` 只在 web/runtime-node 真宿主存在的前提下前进，不回流到 desktop interactive reattach。
