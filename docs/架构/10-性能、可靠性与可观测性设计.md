# 10-性能、可靠性与可观测性设计

## 1. 目标

系统必须同时满足低延迟、高吞吐、长会话稳定、可恢复、可诊断五个目标，并把这些目标转化为可验证的工程门禁。

## 2. 性能目标

| 指标 | 目标 |
| --- | --- |
| 桌面冷启动到可交互 | `< 2s` |
| 新建本地 Session | `< 300ms` |
| 本地输入回显 p95 | `< 16ms` |
| 远程输入回显 p95 | `< 80ms` |
| Session Reattach | `< 1s` |
| Scrollback | `100,000+` 行可用 |
| 大输出场景 | UI 不冻结、路径可诊断 |

## 3. 可靠性设计

- Session 真相必须由 runtime 持有，UI 只持有 attachment。
- `Local Session Daemon` 必须与 UI 进程解耦。
- 渲染失败、桥接降级、窗口关闭都不能等价于 Session 丢失。
- Replay、审计、诊断与热路径要分层，避免大输出拖垮控制面。
- Desktop 与 Server 共享恢复语义，但不共享宿主生命周期实现。
- one-shot local shell execute 不得被当作完整 interactive PTY 性能证据。

## 5. 可观测性设计

### 5.1 Metrics

- Session 创建、Attach、Detach、Reattach、Terminate 成功率
- 输入回显延迟、输出吞吐、Resize 延迟
- connector 成功率
- CLI 启动成功率
- daemon `start / reconnect / stop` 计数
- bridge `ready / degraded` 占比
- 指标维度必须包含 `platformFamily / cpuArch / hostMode / targetType`

### 5.2 Logs

- host 日志
- runtime 日志
- connector 日志
- CLI 诊断日志
- 审计日志
- bridge readiness 失败原因
- Desktop 需带 `Windows / Ubuntu / macOS` 宿主诊断字段；Server 需带 `Ubuntu Server` broker/node 诊断字段

### 5.3 Trace

- 统一 `traceId / sessionId / workspaceId / runtimeNodeId`
- Public API、Manage API、Internal API、Runtime 热路径必须可串联

### 5.4 Diagnostic Bundle

- 版本信息
- 配置快照
- 关键日志
- 崩溃信息
- 指标摘要
- 平台与依赖诊断结果

## 6. 平台性能门禁

- Desktop `P0`：`Windows x64`、`Ubuntu x64`、`macOS arm64`
- Desktop `P1`：`Windows arm64`、`Ubuntu arm64`、`macOS x64`
- Server `P0`：`Ubuntu Server x64 / arm64`
- 任一目标没有构建、启动、交互、恢复、诊断证据，不得宣称“已支持”

## 7. 发布门禁

- 冷启动、输入回显、Reattach 不得明显回退
- 大输出、长会话、弱网、桥接降级 smoke 必须通过
- 关键 Trace / Metric / Log 字段不得缺失

## 8. 当前评估

| 能力 | 标准 | 当前结论 |
| --- | --- | --- |
| Desktop Host + Local Daemon | 最小 readiness、启停、重连、错误可见 | `L3` |
| Terminal 输出稳定性 | 大输出基线 smoke 与可诊断路径 | `L4` |
| Session Recovery | 会话真相、重连、回放、持久化恢复 | `L4` |
| 平台证据门禁 | `P0 / P1` 维度可落到验证矩阵 | `L4` |
