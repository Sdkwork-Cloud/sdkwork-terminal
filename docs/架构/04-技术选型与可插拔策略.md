# 04-技术选型与可插拔策略

## 1. 选型原则

- 桌面交互要像专业 terminal，不能像 Web 控制台拼装品。
- 平台差异必须显式建模，不能靠“一个抽象吃掉所有系统差异”。
- 共核层优先追求契约稳定；适配层优先追求正确性、性能与可观测性。
- AI CLI 集成只做原生托管，不做统一语义层。
- 发布物必须按目标三元组构建，不能以“理论跨平台”代替真实交付。

## 2. 冻结技术栈

| 层 | 方案 | 冻结结论 | 说明 |
| --- | --- | --- | --- |
| Desktop Host | `Tauri 2` | 冻结采用 | 适合薄宿主、自定义 header、窗口控制、权限与 sidecar 交付 |
| Frontend Shell | `React + TypeScript + Vite` | 冻结采用 | 桌面 Shell 与后续 Web Console 共享交互层 |
| Terminal View | `xterm.js` + adapter | 冻结采用 | 只做渲染层，不做业务真相源 |
| Runtime Kernel | `Rust + tokio` | 冻结采用 | 承载 PTY、Session、Replay、Connector、CLI Host |
| VT/ANSI Parser | `vte` 或等价 Rust 解析器 | 冻结采用 | 统一 VT 语义，不让 WebView 解析热路径成为权威 |
| 本地存储 | `SQLite (WAL)` | 冻结采用 | Desktop Session/Replay/Index/Settings 基线 |
| 服务端存储 | `PostgreSQL + Redis + Object Storage` | 冻结采用 | Server Mode 基线 |
| 服务端框架 | `axum / tower` 风格 Rust 栈 | 冻结采用 | 适合 broker/runtime node/control plane |
| 观测 | `tracing + OpenTelemetry` | 冻结采用 | Desktop/Server 统一 trace fields |
| 构建与包管理 | `pnpm workspace + cargo workspace` | 冻结采用 | packages/crates 共核工程基线 |

## 3. 平台适配技术策略

### 3.1 PTY/终端内核

- 平台统一接口：`TerminalKernelPort`、`PtyBackendPort`、`SessionRuntimePort`。
- Windows 后端：以 `ConPTY` 为标准实现。
- Ubuntu/macOS 后端：以 `POSIX PTY` 为标准实现。
- 可以使用 `portable-pty` 或等价库作为工程加速器，但不能把它当成“平台差异消失器”；`ConPTY` 与 `POSIX PTY` 的编码、控制、resize、信号与 shell 启动差异必须在产品层可见、可测、可诊断。

### 3.2 Desktop 流式通信

- 控制通道：`Tauri command/event`。
- 终端热路径：Windows 使用 `Named Pipe`，Ubuntu/macOS 使用 `Unix Domain Socket`。
- 结论：`invoke` 只适合控制面，不适合承载高吞吐终端字节流。

### 3.3 Shell 与环境发现

- Windows：`cmd`、`Windows PowerShell`、`pwsh`、可选 `WSL profile`。
- Ubuntu：`bash`、`zsh`、`fish`。
- macOS：`zsh`、`bash`、`fish`，并补齐 GUI 启动下的登录 shell 环境导入。

### 3.4 CLI 集成

- `Codex`、`Claude Code`、`Gemini`、`OpenCode` 都以“本地已安装 CLI + 启动模板 + 环境白名单 + 进程托管 + 恢复/观测”方式集成。
- 平台只统一 `launch template / session attachment / logging / diagnostics / recovery`，不统一各产品命令语义。

### 3.5 Connector

- SSH：优先标准化本地 OpenSSH/agent/known_hosts 兼容能力，并预留原生协议后端。
- Docker：桌面优先复用本地 `docker` CLI 与 context；服务端允许演进到 Engine API。
- Kubernetes：桌面优先复用 `kubectl` 与 kubeconfig；服务端允许演进到原生 Kubernetes API。
- Remote Runtime：统一 `ExecutionTarget` 契约，不统一底层传输实现。

## 4. 交付技术策略

### 4.1 目标三元组

- Windows：`x86_64-pc-windows-msvc`、`aarch64-pc-windows-msvc`
- macOS：`aarch64-apple-darwin`、`x86_64-apple-darwin`
- Ubuntu/Desktop 与 Ubuntu Server：`x86_64-unknown-linux-gnu`、`aarch64-unknown-linux-gnu`

### 4.2 产物规则

- 每个目标三元组单独构建、单独测试、单独签名、单独发布。
- `externalBin`、sidecar、安装脚本、更新元数据都必须带目标三元组，不允许共享“无架构名”二进制。
- 如需 `universal` 包，只能作为两个单架构包都通过验证后的附加交付物，不能替代单架构标准产物。

## 5. 可插拔边界

- 渲染器可插拔：`xterm.js` 是当前实现，不是对外契约。
- PTY 后端可插拔：`Windows ConPTY` 与 `POSIX PTY` 统一挂在 `PtyBackendPort` 之下。
- Connector 可插拔：SSH / Docker / Kubernetes / Remote Runtime。
- CLI Host 可插拔：每个 AI CLI 独立 adapter。
- 存储可插拔：Desktop `SQLite` 与 Server `PostgreSQL/Redis/Object Storage` 通过 port 解耦。
- 认证可插拔：Desktop 系统凭证库、Server OIDC/SSO。

## 6. 不推荐方案

- 不把 `xterm.js` 直接暴露给业务 feature。
- 不把 `ConPTY` 和 `POSIX PTY` 强行揉成无差别实现。
- 不把 `Tauri IPC invoke` 作为终端热路径。
- 不把 AI CLI 集成改造成统一 AI Provider 抽象层。
- 不把 `docker / k8s` 视为“桌面再打一层包”的延伸。
- 不直接硬耦合 WezTerm/Ghostty 等完整产品内部实现。

## 7. 评估标准

| 评估项 | 合格线 | 领先线 | 当前要求 |
| --- | --- | --- | --- |
| 技术路线稳定性 | 栈明确、边界明确 | 关键热路径和交付路径都可测可替换 | `L4` |
| 平台正确性 | 平台差异显式建模 | 差异可测试、可诊断、可发布 | `L4` |
| Rust 利用度 | PTY/Session/Replay 下沉 Rust | 流式热路径与控制面彻底分离 | `L4` |
| 可插拔设计 | 有 adapter/port | 替换实现不改上层契约 | `L4` |

## 8. 结论

当前最优技术路线不是继续追逐“全平台同构”，而是保持 `Tauri + React` 的高质量桌面壳层、`Rust` 的终端内核与治理能力、以及“共核契约 + 平台适配 + 目标三元组发布”的专业交付纪律。
