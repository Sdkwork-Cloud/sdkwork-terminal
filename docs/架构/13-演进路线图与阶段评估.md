# 演进路线图与阶段评估

## 1. 目标

本文档把架构基线落到实施阶段、阶段目标与退出标准。

## 2. 当前阶段判断

- 当前状态：`L2`，架构冻结中。
- 当前重点：先收敛标准，再进入实现。
- 当前结论：不得跳过 `Session / Protocol / Package Boundary / Delivery Standard` 直接堆功能。

## 3. 路线图

### Phase 0：架构验证

目标：

- Tauri + React + Rust 基础骨架
- Local Session Daemon PoC
- xterm.js Adapter 验证
- `Windows / Ubuntu / macOS` 三平台最小跑通

退出标准：

- 本地 Session 可创建、可输入、可输出、可 Resize
- UI 与 Session Daemon 已解耦
- 基础包结构成型
- `Windows x64`、`Ubuntu x64`、`macOS arm64` 至少形成 `P0` 级最小证据

### Phase 1：专业 Terminal MVP

目标：

- Tabs / Panes / Profiles / Themes / Keybindings
- Search / Clipboard / Scrollback
- Session Center 基础版
- 命令面板与基础 Shell Integration

退出标准：

- 主流 Shell 与关键 TUI 可用
- Session 恢复基础版可用
- 关键性能指标达到基线

### Phase 2：远程与资源目标

目标：

- SSH
- Docker
- Kubernetes
- Resource Center
- Replay 基础版

退出标准：

- 远程目标可稳定创建 Session
- 断线、重连、Detach/Attach 可用
- 资源树与 Session Center 闭环

### Phase 3：AI CLI Native Workspace

目标：

- `Codex / Claude Code / Gemini / OpenCode`
- CLI 检测、模板、诊断、恢复
- 远程/容器/K8s 内 CLI 启动

退出标准：

- 多 CLI 启动链稳定
- CLI Session 纳入统一历史、恢复、诊断体系

### Phase 4：Server Mode

目标：

- Public API / Manage API / Internal API
- Session Broker
- Runtime Node
- Web Console
- Docker / K8s 部署

退出标准：

- 双主态、两包装、共核但不全共享落地
- 多用户、工作区、审计、Broker/Node 协调可用
- `Ubuntu Server x64 / arm64`、Docker、Kubernetes 的主交付链可验证

### Phase 5：企业治理

目标：

- SSO / RBAC / 策略 / 配额 / HA
- 合规审计与商业交付完善

退出标准：

- 可进入团队与企业生产环境

## 4. 阶段评估标准

| 阶段 | 评估重点 |
| --- | --- |
| Phase 0 | 架构是否正确、边界是否稳定 |
| Phase 1 | 专业 Terminal 体验是否成立 |
| Phase 2 | 远程与基础设施场景是否闭环 |
| Phase 3 | CLI 原生集成是否稳定而不越界 |
| Phase 4 | Server Mode 是否真正复用共核契约而不复用错误宿主层 |
| Phase 5 | 治理与商业化是否可交付 |

## 5. 结论

实施顺序必须遵守 `基础内核 -> 终端体验 -> 远程与资源 -> CLI 原生 -> Server Mode -> 企业治理`，不得倒置。
