# 综合评估矩阵与优先级清单

## 1. 目标

本文档用于统一评审口径、版本优先级与长期风险管理。

## 2. 综合评估矩阵

| 维度 | 当前目标 | 当前判断 |
| --- | --- | --- |
| 产品定位 | Terminal-first，边界清晰 | `L4` |
| 架构标准 | 双主态、两包装、共核但不全共享，Host/Shell/Feature/Foundation 清晰 | `L4` |
| 首页形态 | `tabs/custom header + terminal stage` 稳定成立，辅助域默认收起 | `L4` |
| 终端体验 | 对标 Windows Terminal / WezTerm / VS Code Terminal | `L3-L4` |
| 会话模型 | Session-first，可恢复、可回放 | `L4` |
| AI CLI 集成 | CLI 原生、不做语义抽象 | `L4` |
| Server Mode | Day-1 架构进入 | `L4` |
| 性能与可靠性 | 有量化目标与故障演练要求 | `L4` |
| 安全与测试 | 有制度化标准 | `L4` |
| 交付与发布 | 支持桌面、Server、Docker、K8s | `L4` |
| 平台矩阵 | `Windows / Ubuntu / macOS / Ubuntu Server` 与 `x64 / arm64` 有明确证据门禁 | `L4` |

## 4. P0 清单

- 桌面骨架
- Local Session Daemon
- Terminal View Adapter
- 自定义 header / tabs / window controls / terminal stage 首页主链
- Session 状态机
- Local / SSH / Docker / Kubernetes 统一 Session 模型
- AI CLI Native Host
- `packages/sdkwork-terminal-xxx + @sdkwork/terminal-xxx` 分包基线
- Public / Manage / Internal API 基线
- `Windows x64`、`Ubuntu x64`、`macOS arm64`、`Ubuntu Server x64 / arm64` 平台证据

## 5. P1 清单

- Replay 增强
- Shell Integration 增强
- Web Console
- 多节点 Broker
- 自动升级与回滚
- `Windows arm64`、`Ubuntu arm64`、`macOS x64` 平台证据

## 6. P2 清单

- 企业 SSO
- HA 与配额治理
- 受控插件体系
- 高级审计与合规增强

## 7. L1-L5 判定口径

| 等级 | 定义 |
| --- | --- |
| `L1` | 概念级，只有方向，没有标准 |
| `L2` | 架构冻结级，有标准，无实现证据 |
| `L3` | 可用级，主链可运行 |
| `L4` | 产品化级，标准、测试、交付闭环 |
| `L5` | 行业领先级，可规模交付且长期稳定演进 |

## 8. 风险清单

| 风险 | 说明 | 缓解 |
| --- | --- | --- |
| WebView 差异 | 三平台渲染差异明显 | Adapter 隔离 + 平台矩阵测试 |
| xterm.js 上限 | 极端场景可能遇到瓶颈 | 提前隔离渲染契约 |
| 远程连接复杂度 | SSH/K8s/Docker 认证差异大 | 统一 Target 模型，后端双路线 |
| CLI 变化快 | 命令、参数、认证会变化 | 单 CLI 单 Adapter + 契约测试 |
| Server Mode 膨胀 | 易形成第二套系统 | 强制“共核但不全共享”，只允许 server 包装扩展 |
| 平台支持失真 | 文档先于证据宣称已支持 | 强制 `P0 / P1 / 未验证` 三态声明 |
| 首页回摆为 dashboard | 辅助面膨胀吞掉 terminal 主舞台 | 将 `tabs/custom header + terminal stage` 写入验收门禁与回归测试 |

## 9. 结论

当前路线正确，风险集中在 `终端兼容性、连接器复杂度、CLI 适配演进、Server Mode 共核纪律` 四条主线，应持续用契约、测试与发布门禁锁定。
