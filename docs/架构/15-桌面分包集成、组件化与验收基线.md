# 桌面分包集成、组件化与验收基线

## 1. 目标

本文档冻结 Tauri 桌面分包模式、目录命名规则、`package.json.name` 规则、组件化结构与验收基线。分包结构与宿主职责对齐 `claw-studio` 的 workspace 纪律，但产品形态必须保持 terminal-first。

## 2. 命名规则

- 前端目录名统一使用 `packages/sdkwork-terminal-xxx`。
- 前端 `package.json.name` 统一使用 `@sdkwork/terminal-xxx`。
- `apps/desktop` 与 `apps/web` 的包名分别固定为 `@sdkwork/terminal-desktop`、`@sdkwork/terminal-web`。
- Host 包、Shell 包、Feature 包、Foundation 包全部遵守同一职责后缀规则。
- 目录名、包名、导入名必须分别校验，不允许混用或省略作用域。
- 不允许引入 `packages/web-shell`、`packages/desktop-shell` 这类脱离 `sdkwork-terminal-xxx` 规范的目录名。

## 3. 推荐分包清单

### 3.1 Host / Shell

- `@sdkwork/terminal-desktop`
- `@sdkwork/terminal-web`
- `@sdkwork/terminal-shell`

### 3.2 Feature

- `@sdkwork/terminal-workbench`
- `@sdkwork/terminal-sessions`
- `@sdkwork/terminal-resources`
- `@sdkwork/terminal-ai-cli`
- `@sdkwork/terminal-settings`
- `@sdkwork/terminal-diagnostics`

### 3.3 Foundation

- `@sdkwork/terminal-core`
- `@sdkwork/terminal-infrastructure`
- `@sdkwork/terminal-contracts`
- `@sdkwork/terminal-types`
- `@sdkwork/terminal-ui`
- `@sdkwork/terminal-i18n`
- `@sdkwork/terminal-commons`

## 4. Tauri 分包集成原则

- `src-tauri` 只负责宿主桥接、窗口能力、打包与本地桥入口。
- `@sdkwork/terminal-desktop` 只做桌面 Host，不承载 Feature 实现。
- `@sdkwork/terminal-shell` 只负责全局布局、导航、路由与装配。
- Feature 包必须可被桌面端和未来 Web Console 复用。
- 自定义 header、tabs、window controls、drag region 归 `@sdkwork/terminal-shell` 组合；`src-tauri` 只暴露窗口权限与原生控制入口。
- 首页必须保持 `顶部 tabs/custom header + 剩余区域 terminal stage`；Feature 不能通过分包集成把首页改造成 dashboard。
- Tauri CLI 必须通过 workspace 根目录驱动；统一使用 `tools/scripts/run-tauri-cli.mjs` 收口执行目录。
- `beforeDevCommand / beforeBuildCommand` 必须使用 root-relative `apps/desktop` 路径，不允许依赖 `src-tauri` 相对目录假设。
- `src-tauri` 最小骨架必须包含 `Cargo.toml`、`build.rs`、`src/main.rs`、`src/lib.rs`、`tauri.conf.json`、`capabilities/default.json`。

## 5. 允许依赖矩阵

| From | Allowed | Forbidden |
| --- | --- | --- |
| Host | Shell、Infrastructure | Feature 业务内部实现 |
| Shell | Feature、UI、Contracts | Rust 平台细节直连 |
| Feature | Core、Infrastructure、Contracts、UI | 其他 Feature 内部路径 |
| UI | Types、Design Token | Runtime / Connector / DB |
| `src-tauri` | Desktop Host Bridge | Feature 页面与状态 |
| `apps/web` | Shell、Feature、Contracts | 桌面私有 bridge / window API |

## 6. 包内结构基线

每个 Feature 包推荐结构：

```text
src/
  application/
  components/
  services/
  models/
  hooks/
  index.ts
```

规则：

- 只能从 `index.ts` 暴露公共能力。
- 不允许跨包引用 `src/**` 内部路径。
- 页面层只负责表达，不承担平台与协议细节。

## 7. 组件化规则

- 单组件单职责。
- 大状态拆域，不做巨型全局 store。
- Terminal Adapter 独立封装。
- 平台能力集中在 infrastructure，不散落在页面。
- `Sessions / Resources / AI CLI / Settings / Diagnostics` 只能以 drawer、overlay、palette、modal 方式挂入 terminal shell。

## 8. CI / Review 门禁

- 校验目录名是否符合 `packages/sdkwork-terminal-xxx`。
- 校验 `package.json.name` 是否符合 `@sdkwork/terminal-xxx`。
- 校验跨包引用是否只来自包根。
- 校验 `src-tauri` 是否出现业务页面与业务状态。
- 校验 Host/Shell/Feature/Foundation 依赖方向。
- 校验首页仍保持 `tabs/custom header + terminal stage`，无 dashboard 回归。

## 9. 验收基线

- Host/Shell/Feature/Foundation 依赖方向正确。
- 所有包都有清晰公共出口。
- 任一 Feature 包都不直接依赖 `src-tauri`。
- `@sdkwork/terminal-desktop` 更换为 `@sdkwork/terminal-web` 时，除宿主能力外不应重写业务逻辑。
- `@sdkwork/terminal-shell` 组合出的首页仍是 terminal 首页，而不是多业务面板首页。

## 10. 评估标准

| 评估项 | 合格线 | 领先线 | 当前判断 |
| --- | --- | --- | --- |
| 分包纪律 | 目录名与包名规则同时成立 | 分包边界可用脚本自动审计且与导入名全一致 | `L4` |
| Tauri 集成纯度 | `src-tauri` 保持薄层 | 可平滑复用到 Web Console | `L4` |
| Terminal 首页纯度 | 分包集成不破坏 tabs/header + terminal stage | 辅助域全部可收起且不抢主舞台 | `L4` |

## 11. 结论

桌面分包标准已经明确收口为三条硬规则：`packages/sdkwork-terminal-xxx` 负责目录标准，`@sdkwork/terminal-xxx` 负责包名与导入标准，`tabs/custom header + terminal stage` 负责首页产品标准；三者缺一不可。
