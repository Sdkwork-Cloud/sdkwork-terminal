# 2026-04-11 v0.2.40 Windows Interactive Probe Alignment

## Scope

- Windows desktop local-shell runtime defaults
- Windows runtime-node interactive PTY defaults
- Split command input plus standalone `Enter` behavior
- Regression coverage for host-side probe configuration

## Changed

- `src-tauri/src/lib.rs`
  - Desktop local-shell runtime now uses synthetic probe responses on Windows and keeps transparent mode on non-Windows hosts.
- `crates/sdkwork-terminal-runtime-node/src/host.rs`
  - Runtime-node host now uses the same Windows-only synthetic probe default for interactive PTY sessions.
- `crates/sdkwork-terminal-pty-runtime/tests/local_shell_session_runtime.rs`
  - Added an integration test that writes command text first and sends `\r` separately, verifying real shell execution still completes.
  - Added an integration test that writes command text first, then applies `DEL` backspace edits and `Enter`, verifying xterm-style correction stays valid before execution.
- `crates/sdkwork-terminal-runtime-node/tests/runtime_node_host_test.rs`
  - Added the same split-input `Enter` verification through the runtime-node host layer.
  - Added the same `DEL` backspace correction verification through the runtime-node host layer.
- `tests/runtime-host-probe-mode.test.ts`
  - Added a source-level regression that locks the Windows probe-mode contract for desktop and runtime-node hosts.

## Fixed

- Fixed a Windows interactive PTY mismatch where direct local runtime tests used synthetic probe responses but real desktop/server host entrypoints still used transparent mode.
- Fixed runtime-node host failing the real-world pattern “type command first, send `Enter` later” on Windows.
- Reduced the risk that the desktop Tauri host would exhibit the same Windows input instability as the failing runtime-node path.
