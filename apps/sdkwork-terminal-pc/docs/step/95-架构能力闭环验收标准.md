# 架构能力闭环验收标准

## 1. 判定原则

能力闭环不以“有代码”为准，而以“有实现 + 有测试 + 有平台证据 + 有发布/回滚口径”为准。

## 2. 能力等级定义

- `L0`：仅规划，未落地
- `L1`：骨架成立，可编译/可装配
- `L2`：单主机场景可用
- `L3`：跨平台或跨目标主链可用
- `L4`：具备恢复、观测、安全与回滚口径
- `L5`：具备正式商业发布证据

## 3. 必须闭环的能力

- Desktop Host / Window Chrome / Permission Bridge
- Terminal Core / Renderer Adapter / PTY Backend
- Session / Replay / Recovery
- ExecutionTarget / Connector Runtime
- AI CLI Native Host
- Workbench 辅助面
- Server Mode / Broker / Runtime Node
- Persistence / Observability / Security / Test Matrix
- Install / Deploy / Release / Upgrade / Rollback

## 4. 最低达标线

| 能力 | 主 Step | 最低目标 | 必要补充 |
| --- | --- | --- | --- |
| Desktop Host / Window Chrome | `02`、`04`、`12` | `L4` | 必须有 `Windows / Ubuntu / macOS` 启动、自定义 header、window controls 与窗口证据 |
| Terminal Core / Renderer / PTY | `05` | `L4` | 必须有 `ConPTY` 与 `POSIX PTY` 证据 |
| Session / Replay / Recovery | `06` | `L4` | 必须有桌面三平台恢复与回放证据路径、`ubuntu-server` runtime-node recovery 基线与仓库级 smoke matrix |
| ExecutionTarget / Connectors | `07` | `L4` | 必须有桌面三平台工具链与 Server remote runtime 证据 |
| AI CLI Native Host | `08` | `L4` | 必须有 CLI 启动、恢复、日志与失败诊断证据 |
| Workbench 辅助面 | `09` | `L4` | 必须证明首页仍为 `tabs/custom header + terminal stage`，辅助面默认收起且不抢主舞台 |
| Server Mode | `10` | `L4` | 必须有 Ubuntu Server `x64 / arm64` 证据 |
| Persistence / Security / Test Matrix | `11` | `L4` | 必须有 `P0` 自动化矩阵证据 |
| Install / Release / Rollback | `12` | `L5` | 必须有目标三元组发布物、签名/公证、回滚证据 |

## 5. 平台支持声明标准

### 5.1 Desktop P0

- Windows `x64`
- Ubuntu `x64`
- macOS `arm64`

### 5.2 Desktop P1

- Windows `arm64`
- Ubuntu `arm64`
- macOS `x64`

### 5.3 Server P0

- Ubuntu Server `x64`
- Ubuntu Server `arm64`

规则：

- 通过 `P0` 才能进入正式发布。
- `P1` 未通过时，只能宣称“核心平台已完成”，不能宣称“全平台全架构完成”。
- 没有证据的目标只能标记为“未验证”。

## 6. 闭环标准

每项能力至少满足：

- 有明确主 Step
- 有共核契约与平台适配边界
- 有代码落点与测试入口
- 有目标平台/架构证据
- 有失败回滚路径
- 有对应架构章节回写
- 有 review 与 release 归档
- 有明确当前等级 `L0-L5`
- 若涉及桌面首页，还必须有“无 dashboard 回归”证据

## 7. Step 完成判定

- 一个 Step 只有在其主能力达到目标等级、平台证据到位、`review + release` 证据齐全后，才算完成。
- 一个波次只有在本波次的 `P0` 验收通过后，才算闭环。
- 如果某能力仍停留在 `L1/L2`，不得在 Step 13 宣称“发布就绪”。
- `Step 09` 若不能证明首页仍保持 `tabs/custom header + terminal stage`，视为未完成。

## 8. 结论规则

- 任一核心能力未闭环，不得宣称“架构已落地完成”。
- 任一平台/架构缺少证据，不得宣称“已支持”。
- 任一交付链缺少安装、升级或回滚证据，不得视为 `L5`。
- 任一文档仍用“四态全共享”描述当前架构，视为架构语义未闭环。
- 任一首页实现回摆为 dashboard、多业务主面板或大卡片入口页，视为产品语义未闭环。
