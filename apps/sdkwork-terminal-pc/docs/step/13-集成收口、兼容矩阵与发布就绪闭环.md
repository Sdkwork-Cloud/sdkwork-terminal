# Step 13 - 集成收口、兼容矩阵与发布就绪闭环

## 1. Step Card

| 项 | 内容 |
| --- | --- |
| 执行模式 | 强串行 |
| 前置 | `00-12` |
| 主写入范围 | `docs/step/13-*`、`docs/架构/README`、`docs/架构/13`、`docs/架构/14`、`docs/review/`、`docs/release/` |
| 执行输入 | 所有 Step 文档、矩阵结果、release 候选产物、review 证据 |
| 本步非目标 | 不新增新功能；不掩盖未完成平台证据 |
| 最小输出 | 总矩阵结论、RC 决策、残余风险、下一轮 backlog |

## 2. 设计

- 总集成只以“能力兑现 + 平台证据 + 发布链证据”判定，不以“代码量”判定。
- 终态结论必须区分：`已通过 P0`、`已通过 P1`、`未验证`、`明确不支持`。
- 发布就绪必须同时覆盖 terminal 体验、平台适配、Server Mode、交付链与安全证据。

## 3. 实施落地规划

1. 汇总 `00-12` 的能力、测试、review、release 证据。
2. 执行总矩阵：`Windows / Ubuntu / macOS / Ubuntu Server`、`x64 / arm64`、shell/TUI/connector/AI CLI/upgrade/rollback。
3. 生成 RC 候选与平台支持清单。
4. 冻结残余风险、降级策略、已知限制与下一轮 backlog。

## 4. 测试计划

- 全量 smoke 回归
- P0 平台总回归
- P1 平台扩展回归
- 发布候选安装、升级、回滚验证
- Server/Docker/K8s 部署候选验证

## 5. 结果验证

- 可以明确回答“这版支持哪些平台/架构，证据在哪，风险是什么”。
- 可以明确回答“desktop/server 共核到哪一层，哪些实现仍必须分平台维护”。
- 可以明确回答“是否能对外宣称全平台完成，若不能，缺口在哪里”。

## 6. 检查点

- `CP13-1`：各 Step 能力与证据归档完成。
- `CP13-2`：P0 总矩阵全部通过。
- `CP13-3`：P1 状态已清晰标注为通过/未通过/未验证。
- `CP13-4`：RC 决策、残余风险与下一轮 backlog 冻结。

## 7. 串并行策略

- 本步必须由单一集成 owner 收口。
- 允许并行跑 smoke、compat、install、upgrade、rollback、server deploy，但所有结论只能统一归档到本 Step。
- 不允许在总验收阶段继续追加新需求或重构范围。

## 8. 风险与回滚

- 风险：局部功能完成被误判为“全平台发布就绪”。
- 回滚：按矩阵结果回滚到对应 Step 修复，不允许在 Step 13 中掩盖缺口。

## 9. 完成定义

- 已形成可对外使用的 release candidate 结论，或形成明确、可执行的回退/补齐计划。
- 架构、Step、review、release 文档已统一口径。
- 平台支持声明与证据完全一致。

## 10. 下一步准入条件

- 若 `P0` 未通过，必须回退到对应 Step 修复。
- 若要宣称“全平台全架构完成”，`P1` 也必须通过。
