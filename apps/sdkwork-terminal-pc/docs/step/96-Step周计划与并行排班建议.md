# Step 周计划与并行排班建议

## 1. 推荐节奏

- 周初：冻结本周目标、依赖、写入范围
- 周中：按并行车道推进实现与测试
- 周末：统一收口、统一验证、统一复盘

## 2. 推荐角色

- 集成 Owner
- 架构 / 契约 Owner
- 前端 / Workbench Owner
- Rust Runtime Owner
- Connector / CLI Owner
- Server / Control Plane Owner
- Verification Owner

## 3. 推荐团队配置

### 3-4 人基线队形

- 适合：仓库起步期、`00-04`
- 编排：`1` 主脊柱 + `2` 实现 + `1` 验证
- 策略：偏串行，只在 `05-08` 做有限并行

### 5-6 人推荐队形

- 适合：最快且风险可控的默认方案
- 编排：`1` 主脊柱 / 集成 + `4` 实现车道 + `1` 验证
- 策略：`05-08`、`09-12` 可以形成稳定并行

### 7-9 人强化队形

- 适合：希望压缩周期，但能承担更高协作成本
- 编排：`1` 架构 / 契约 + `1` 集成 + `5-6` 实现 + `1` 验证
- 策略：只在 `03`、`04`、`06` 已冻结后开启高并行

## 4. 周内排班建议

- 周一：冻结波次目标、检查点、共享契约、写入范围
- 周二至周四：按车道推进实现、单测、集成 smoke
- 周五：统一集成、统一验证、统一回写 `docs/架构` 与 `docs/step`

## 5. 每日节奏建议

- 上午：同步阻塞、冻结当天主写文件
- 下午：实现与本地验证
- 收工前：提交变更摘要、测试结果、风险点、是否可收口

## 6. 排班原则

- 高风险车道优先配置独立验证。
- 同一人不同时主写两个强耦合车道。
- 每周必须保留收口窗口，不连续无限并行。
- 契约 owner 不同时承担高频 UI 交互车道。
- 集成 owner 不承担跨三条以上实现车道的主写责任。

## 7. 快速完成建议

- `00-04` 由最强 owner 串行推进，其他人只做影子验证和模板准备。
- `05-08` 采用 `Runtime / Connector / CLI / Verification` 四车道并行。
- `09-12` 采用 `Workbench / Server / Governance / Delivery / Verification` 五车道并行。
- `13` 只保留集成 owner 和验证 owner 统一收口，避免发布结论分叉。

## 8. 最短编排模板

- 波次 A：`S + V`，冻结骨架、契约、最小闭环
- 波次 B：`R + C + V`，先做 runtime/session，再接 connector/CLI
- 波次 C：`W + P + G + V`，工作台、服务端、治理并行推进
- 波次 D：`G + V + S`，并行产物交付，串行发布评审

执行要点：

- 每波次开始先冻结共享契约
- 每波次中段统一做一次集成 smoke
- 每波次结束统一做 `91 / 93 / 95 / 97` 审计与回写

## 9. 多子 Agent 最短模板

- `S`：主 agent，负责主脊柱、共享契约冻结、最终收口
- `R/C/W/P/G`：每车道 `1` 个实现 agent，主写各自目录
- `V`：独立验证 agent，只写测试、脚本、审计与证据
- 每波次结束由 `S` 汇总 `98` 交付包并统一决定是否进入下一波次
