# Step 00 - 总实施原则与执行门禁

## 1. Step Card

| 项 | 内容 |
| --- | --- |
| 执行模式 | 强串行 |
| 前置 | 无 |
| 主写入范围 | `docs/step/`、`docs/架构/`、`docs/release/`、`scripts/check-*` |
| 执行输入 | `docs/架构/README`、`02`、`11`、`12`、`16`、`17`、`AGENTS.md` |
| 本步非目标 | 不实现终端功能；不新增业务页面；不修改发布版本号 |
| 最小输出 | 平台矩阵、支持声明规则、证据口径、检查点命名、回写规则 |

## 2. 设计

- 冻结“共核但不全共享”原则：`desktop/server` 共享契约与模型，不共享宿主与发布链实现。
- 冻结支持矩阵：`Windows / Ubuntu / macOS` + `x64 / arm64`；`Ubuntu Server x64 / arm64`。
- 冻结发布矩阵：`P0` 是首发阻塞矩阵，`P1` 是扩展阻塞矩阵。
- 冻结支持声明规则：没有目标证据，只能写“未验证”，不能写“支持”。

## 3. 实施落地规划

1. 冻结平台/架构矩阵、目标三元组与发布命名。
2. 冻结 `desktop / server / docker / k8s` 的角色边界：双主态、两包装。
3. 冻结“控制通道 vs 终端热路径”分离规则。
4. 冻结 Step 文档模板、检查点命名、review/release 证据口径。
5. 冻结 changelog 回写规则：只写 `Unreleased`，不随手改版本号。

## 4. 测试计划

- 审核 `docs/架构/` 与 `docs/step/` 是否统一采用“共核但不全共享”表述。
- 审核所有后续 Step 是否明确写出平台/架构差异与目标证据。
- 审核脚本、文档与命名规则是否一致。

## 5. 结果验证

- 后续任一步都能明确回答“改哪里、在哪个平台生效、怎么测、怎么回滚”。
- 任何“跨平台支持”都能追溯到矩阵、证据和发布元数据。
- 任何“桌面/服务端共核”都不会再被误解为“宿主实现完全复用”。

## 6. 检查点

- `CP00-1`：双主态、两包装、共核但不全共享的原则冻结。
- `CP00-2`：`Windows / Ubuntu / macOS / Ubuntu Server` 与 `x64 / arm64` 矩阵冻结。
- `CP00-3`：`P0 / P1` 支持声明与发布门禁冻结。
- `CP00-4`：Step 模板、检查点、证据口径冻结。
- `CP00-5`：changelog、架构回写与 release 证据规则冻结。

## 7. 串并行策略

- 本步必须串行完成。
- 允许先并行整理规则草案，但最终只能由单一 owner 收口。
- 本步未完成前，不得大规模并行推进平台适配、连接器或发布链。

## 8. 风险与回滚

- 风险：仍用“四态全共享”的错误预期指导实现，导致 PTY、权限、发布链和 Server Mode 混乱。
- 回滚：以本文件为准重写对应 Step，不允许在后续 Step 中自行创造例外。

## 9. 完成定义

- 所有后续 Step 都共享同一套平台矩阵、支持声明、证据与回写规则。
- 任何人都不能再把“理论可跨平台”写成“已支持跨平台”。

## 10. 下一步准入条件

- Step 01 必须按本文件的矩阵、门禁和证据口径开展差距审计。
