# 波次里程碑与阶段总验收矩阵

## 1. 波次 A：基础与骨架

- 包含：`00-04`
- 总验收：
  - workspace 与 Tauri 骨架成立
  - 契约层成立
  - 桌面最小主链可启动
  - `Desktop Host + Local Daemon` 达到 `L3`

快速建议：

- 由单一主脊柱 owner 串行打穿
- 其他成员只做验证脚本、模板、审计材料准备

## 2. 波次 B：终端与运行时核心

- 包含：`05-08`
- 总验收：
  - Terminal 主舞台可用
  - Session / Replay / Recovery 成立
  - Desktop 三平台 recovery 证据路径与 `ubuntu-server` runtime-node recovery baseline 成立
  - Connectors 与 AI CLI 主链成立
  - `Terminal Core`、`Session` 达到 `L4`
  - `Connectors`、`AI CLI Native Host` 至少达到 `L3`

快速建议：

- 采用 `Runtime / Connector / CLI / Verification` 四车道并行
- 每周至少一次统一 smoke 和 replay/reattach 回归

## 3. 波次 C：工作台与服务端

- 包含：`09-11`
- 总验收：
  - 桌面工作台成立
  - Server Mode 主链成立
  - 存储、观测、安全、测试闭环成立
  - `Workbench` 达到 `L4`
  - `Server Mode` 至少达到 `L3`
  - `Observability / Security / Test Matrix` 达到 `L4`

快速建议：

- 采用 `Workbench / Server / Governance / Verification` 四主车道
- 把 `Public/Manage/Internal/Runtime Stream` 作为统一集成窗口强制收口项

## 4. 波次 D：交付与发布

- 包含：`12-13`
- 总验收：
  - 各交付形态可构建
  - 升级/回滚可验证
  - 发布就绪结论成立
  - `Install / Deploy / Release / Upgrade / Rollback` 达到 `L5`
  - 总能力矩阵通过 `13` 的发布评审

快速建议：

- `12` 可按 Desktop / Docker / K8s / Release 四子车道并行
- `13` 只允许一个集成 owner 出最终发布结论

## 5. 波次总规则

- 任一波次未通过总验收，不得开启下一波次的新能力开发。
- 任一波次内如果出现 `B4`，必须停止推进并回到该波次总复盘。
- 最快路径是 `A 串行 -> B 并行 -> C 并行 -> D 并行子车道 + 串行发布结论`。
