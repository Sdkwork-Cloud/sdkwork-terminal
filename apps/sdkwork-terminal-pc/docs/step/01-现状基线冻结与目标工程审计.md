# Step 01 - 现状基线冻结与目标工程审计

## 1. 目标与范围

目标：

- 明确当前仓库与目标工程结构之间的差距。
- 冻结“先做什么、不做什么、往哪落目录”的审计结论。

当前事实：

- 当前仓库以 `README.md + docs/*` 为主，工程骨架尚未进入主分支。
- 尚无 `packages/`、`crates/`、`src-tauri/`、`deployments/`、`tools/`。

非目标：

- 不在本步创建实际功能代码。

## 2. 架构对齐

- `docs/架构/03-模块规划与边界.md`
- `docs/架构/04-技术选型与可插拔策略.md`
- `docs/架构/15-桌面分包集成、组件化与验收基线.md`

## 3. 写入范围

创建或修改：

- `docs/step/01-现状基线冻结与目标工程审计.md`
- `docs/review/` 下的差距矩阵、风险清单、审计记录
- 受影响的 `docs/架构/03`、`04`、`15`

## 4. 设计

- 审计输出必须覆盖目录、包名、crate 名、进程角色、协议面、测试链、交付链。
- 审计结论必须形成“目标目录树 + 差距矩阵 + 实施优先级”。

## 5. 实施落地规划

- 审计当前根目录与文档，生成目标目录规划：
  - `apps/desktop`
  - `apps/web`
  - `packages/*`
  - `crates/*`
  - `src-tauri/`
  - `deployments/docker`
  - `deployments/k8s`
  - `tools/`
- 明确 `packages/` 目录命名统一为 `sdkwork-terminal-xxx`，`package.json.name` 统一为 `@sdkwork/terminal-xxx`。
- 明确 Rust crate、服务端组件、桌面宿主和本地 daemon 的落点。
- 明确 `desktop / server` 是主态，`docker / k8s` 只是 `server` 的交付装配，不是独立业务态。

## 6. 测试计划

- 对照 `docs/架构/03`、`15` 检查目录规划是否完整。
- 检查每一类能力是否都有未来目录落点。

## 7. 结果验证

- 产出差距矩阵，明确“已存在 / 待创建 / 高风险 / 依赖前置”。
- 可回答每个架构能力未来写入哪里。

最小证据：

- 差距矩阵 + 目标目录树 + 风险清单 + 回滚说明

能力兑现：

- 兑现 `docs/架构/03`、`15` 中“目录、分包、crate、进程角色可映射”的要求。
- 为后续所有 step 建立唯一目录落点和写入边界。

架构回写：

- `docs/架构/03-模块规划与边界.md`
- `docs/架构/15-桌面分包集成、组件化与验收基线.md`

快速执行建议：

- 串行先做目标目录树和包名冻结。
- 审计结果形成单一矩阵后，再允许 `02` 的骨架车道并行启动。

## 8. 检查点

- `CP01-1`：目标目录树冻结。
- `CP01-2`：包名、crate 名、进程角色冻结。
- `CP01-3`：差距矩阵与实施优先级冻结。

## 9. 串并行策略

- 本步必须串行。
- 本步结束后，`02` 才允许启动。

## 10. 风险与回滚

- 风险：目录规划与架构文档不一致。
- 回滚：回写 `docs/架构/03`、`15` 后重新审计。

## 11. 完成定义

- 能明确回答“每个能力、每个包、每个 crate、每个服务以后放哪里”。
- 目标状态：审计基线、目录规划与实施优先级固定，不参与 `L` 等级判定。

## 12. 下一步准入条件

- `02` 的骨架创建范围和命名已无歧义。
