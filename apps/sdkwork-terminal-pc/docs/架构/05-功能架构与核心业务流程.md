# 05-功能架构与核心业务流程

## 1. 功能架构总览

产品能力围绕七个域构建：

- `Terminal Workbench`
- `Session Center`
- `Resource Center`
- `AI CLI Center`
- `Profiles & Settings`
- `Diagnostics Center`
- `Server Governance`

统一原则：

- 首页主舞台永远是 `tabs/header + terminal stage`。
- `Resources / Sessions / Settings / Diagnostics` 都是围绕 terminal 主链服务的辅助域。
- `desktop` 与 `server` 共享功能语义，不共享宿主实现与交付链实现。

## 2. 当前事实

- 当前仓库仍处于架构冻结与主链逐步兑现阶段，功能域是正式目标结构，不是“全部已完成”的声明。
- 当前最关键的是先把 `Workbench / Session / Resource / CLI / Server` 五条主链定义稳定，再进入平台化实现。
- `Step 07` 已建立 Resource Center 派生模型、toolchain smoke、launch/probe bridge、session index 与 replay 读链。
- 当前桌面端已能从资源目标创建 launch/probe 读侧证据，但远程 interactive attach / exec 主链仍未闭环。
- 当前本地 one-shot shell execute 已可用，但不能等同于“完整本地交互式终端内核”。

## 3. 功能域分解

### 3.1 Terminal Workbench

职责：

- tabs、pane、profile、theme、keybinding、command palette、search、scrollback、selection。

平台差异：

- Windows：对齐 `ConPTY`、窗口控制、IME、剪贴板、路径语义。
- Ubuntu：对齐 `POSIX PTY`、Wayland/X11、字体与 locale。
- macOS：对齐 `POSIX PTY`、登录 shell 环境、快捷键、签名后的生产运行行为。

### 3.2 Session Center

职责：

- Session 生命周期、attach/detach/reattach、replay、history、ack、diagnostics。

统一要求：

- Session 是一等实体，Attachment 只是句柄。
- 任何平台下 UI 都不能成为 Session 真相源。

### 3.3 Resource Center

职责：

- SSH、Docker、Kubernetes、Remote Runtime 的目标发现、状态派生、launch/probe。

统一要求：

- Resource Center 只持有目标视图，不持有 Session 真相。
- 三平台桌面必须能各自证明 `ssh / docker / kubectl` 工具链与错误诊断链。

### 3.4 AI CLI Center

职责：

- `Codex / Claude Code / Gemini / OpenCode` 的发现、版本、认证状态、模板、启动、恢复、诊断。

统一要求：

- 保持原生命令、原生参数、原生认证与原生输出。
- 只统一托管与恢复，不统一 AI 语义。

### 3.5 Server Governance

职责：

- 用户、工作区、broker、runtime node、审计、策略、发布治理、Web Console。

统一要求：

- 官方运行平台是 `Ubuntu Server x64 / arm64`。
- `docker / k8s` 只包装 `server`，不复制业务流程。

## 4. 核心流程一：桌面启动与恢复

```text
Desktop App Start
-> Host Bootstrap
-> Ensure Local Daemon
-> Load Config + Workspace Snapshot
-> Restore Layout
-> Reattach Recoverable Sessions
-> Enter Ready State
```

要求：

- 宿主启动与 Session 恢复必须解耦。
- 布局恢复失败不得破坏后台 Session。
- 三平台都必须提供 Host Bootstrap 与 Daemon Bridge 证据。

## 5. 核心流程二：新建本地或远程 Session

```text
User Select Profile / Target
-> Resolve ExecutionTarget
-> Validate Runtime Capability
-> Create Session
-> Spawn PTY or Connector-backed Runtime
-> Attach Terminal View
-> Persist Session / Replay Metadata
```

要求：

- `local-shell / ssh / docker-exec / kubernetes-exec / remote-runtime / cli-native` 都必须映射到统一 Session 创建流程。
- 本地 shell：
  - Windows 走 `ConPTY`
  - Ubuntu/macOS 走 `POSIX PTY`
- 失败必须记录目标、平台、命令、阶段、认证、退出码与可诊断上下文。

## 6. 核心流程三：AI CLI 启动

```text
User Choose CLI Template
-> Detect Binary + Version + Auth State
-> Resolve Working Directory / Env / Args
-> Resolve Target(Local/SSH/Docker/K8s)
-> Spawn CLI Session
-> Tag Session As CliNative
-> Persist Launch Metadata
-> Allow Detach / Resume / Replay
```

要求：

- Windows、Ubuntu、macOS 的 CLI 发现逻辑必须考虑不同 shell、PATH、登录方式与安装位置。
- 启动前后都必须记录二进制版本、目标环境、工作目录、模板、退出结果。

## 7. 核心流程四：Detach / Reattach / Replay

```text
User Detach
-> Session Runtime Keeps Ownership
-> UI Releases Attachment
-> Session Continues Running
-> User Reattach or Replay Later
```

要求：

- `Detach` 不能等价于关闭 Session。
- `Reattach` 必须以 Session 真相源为准，不以旧 UI 缓存为准。
- `Replay` 与 `Reattach` 必须分离：前者读历史，后者回到活会话。
- Desktop 与 Server 都必须共享同一语义。

## 8. 核心流程五：Server Mode 托管

```text
Desktop/Web Client
-> Public API / Auth
-> Session Broker
-> Runtime Node
-> Connector / CLI Host / PTY
-> Stream Back by WS/Binary Channel
```

要求：

- `server` 必须复用与桌面一致的 Session 语义。
- `runtime node` 官方主机是 `Ubuntu Server`。
- `docker / k8s` 仅替换交付与编排，不替换业务流程。

## 9. 核心流程六：配置变更生效

```text
Settings / Profile Update
-> Validate Schema
-> Persist Config
-> Notify Runtime or Rebuild Derived Snapshot
-> Apply to New Sessions
-> Selective Apply to Running Sessions If Safe
```

要求：

- 配置修改必须区分 `立即生效`、`新会话生效`、`需重建连接` 三类。
- 任一写操作必须有读回校验路径。
- Desktop 与 Server 的配置模型一致，但生效机制可按宿主差异实现。

## 10. 核心流程七：诊断与问题闭环

```text
User Open Diagnostics
-> Aggregate Host / Runtime / Connector / CLI Logs
-> Collect Metrics + Crash + Config Snapshot
-> Produce Diagnostic Bundle
-> Support Export / Upload / Audit
```

要求：

- Desktop 诊断包必须包含平台、架构、目标三元组、bridge/daemon 摘要。
- Server 诊断包必须包含 broker/node、租约、部署与审计摘要。

## 11. 核心流程八：升级与回滚

```text
Release Arrives
-> Preflight / Readiness
-> Backup Critical Metadata
-> Upgrade Binary / Assets / Schema
-> Smoke / Verify
-> Success or Rollback / Repair
```

要求：

- Windows、Ubuntu、macOS 各自拥有独立安装包、签名/公证与升级验证。
- `Ubuntu Server x64 / arm64`、Docker、Kubernetes 拥有独立交付链，但共享 release metadata。
- 不允许“交付包装不同”演变成“升级语义不同”。

## 12. 评估标准

| 评估项 | 合格线 | 领先线 | 当前要求 |
| --- | --- | --- | --- |
| 功能域完整度 | 覆盖 Terminal、Session、Resources、CLI、Server | 同时覆盖恢复、诊断、交付 | `L4` |
| 流程闭环度 | 新建、恢复、AI CLI、远程、Server 均闭环 | 任一关键流程均可诊断、可回读、可验收 | `L4` |
| 平台化程度 | 流程明确映射 `Windows / Ubuntu / macOS / Ubuntu Server` | 支持声明完全证据化 | `L4` |

## 13. 结论

真正的功能架构不是简单把能力清单列出来，而是把每条主链都拆成“统一语义 + 平台实现 + 发布证据”的可执行流程。
