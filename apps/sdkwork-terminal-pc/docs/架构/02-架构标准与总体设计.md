# 02-架构标准与总体设计

## 1. 总体目标

总体架构必须同时满足五个目标：

1. 对标 `Windows Terminal` 的专业交互与终端心智。
2. 同时支持 `Windows / Ubuntu / macOS` 与 `x64 / arm64`。
3. 在桌面端提供最佳本地交互体验，在服务端提供可托管的远程运行能力。
4. 统一 `Session / Replay / ExecutionTarget / CLI Host / Release Manifest` 共核模型。
5. 支撑安装、部署、升级、回滚与商业化交付。

## 2. 总体架构形态

### 2.1 双主态、两包装

```text
desktop
  -> Tauri Host / Custom Header / Native Permission Bridge
  -> Shell UI / Tabs / Terminal Stage
  -> Desktop Control Bridge + Stream Channel
  -> Rust Terminal Kernel
  -> Platform PTY Adapter (Windows ConPTY | POSIX PTY)
  -> Local Storage / Diagnostics

server
  -> Web Console
  -> Control API / Session Broker
  -> Runtime Node
  -> POSIX PTY / Remote Connector Runtime
  -> PostgreSQL / Redis / Object Storage

docker / k8s
  -> server 的交付包装与运维编排
```

结论：`desktop` 与 `server` 是产品主态，`docker / k8s` 只是服务端交付态，不是对等业务态。

### 2.2 共核但不全共享

| 层 | 是否共核 | 标准 |
| --- | --- | --- |
| 产品心智、导航语义、tab/session 模型 | 是 | `tab = session attachment`，主体永远是 terminal window |
| `Session / Replay / ExecutionTarget / Connector / CLI Launch` 契约 | 是 | 统一模型、统一错误语义、统一事件族 |
| 终端解析、scrollback、search、recovery 语义 | 是 | Rust runtime 统一实现或统一接口 |
| 桌面窗口宿主、标题栏、窗口控制、权限 | 否 | 按 `Windows / Ubuntu / macOS` 分别适配 |
| `ConPTY / POSIX PTY`、shell 发现、环境注入 | 否 | 平台差异显式建模，不做假统一 |
| Server broker / runtime node / mTLS / 多租户 | 否 | 仅存在于 `server` 交付形态 |
| 安装器、签名、公证、镜像、Helm | 否 | 仅共享 release metadata，不共享实现 |

## 3. 平台与架构标准

### 3.1 Desktop 官方支持范围

| 平台 | 架构 | 标准定位 |
| --- | --- | --- |
| Windows | `x64` | 首发阻塞矩阵 |
| Windows | `arm64` | 扩展阻塞矩阵 |
| Ubuntu | `x64` | 首发阻塞矩阵 |
| Ubuntu | `arm64` | 扩展阻塞矩阵 |
| macOS | `arm64` | 首发阻塞矩阵 |
| macOS | `x64` | 扩展阻塞矩阵 |

### 3.2 Server 官方支持范围

| 平台 | 架构 | 标准定位 |
| --- | --- | --- |
| Ubuntu Server | `x64` | 首发阻塞矩阵 |
| Ubuntu Server | `arm64` | 首发阻塞矩阵 |

### 3.3 CPU 与产物规则

- 每个目标三元组必须独立构建、独立测试、独立签名、独立发布。
- 不允许以“可交叉编译”替代“可运行”。
- 不默认提供“单包全平台”或“单包双架构”幻想；如果后续需要 `universal` 包，也必须建立在两个独立目标已分别通过验证之后。

## 4. 运行时控制平面

### 4.1 五个平面

- 交互平面：tabs、终端窗口、命令面板、搜索、设置、诊断。
- 契约平面：`Session`、`Replay`、`ExecutionTarget`、`CLI Host`、错误体、事件流。
- 执行平面：PTY、Connector、CLI 托管、replay、recovery、broker、runtime node。
- 治理平面：权限、审计、配置、策略、用户/工作区、诊断导出。
- 交付平面：安装、升级、回滚、镜像、Helm、SBOM、签名、公证。

### 4.2 Session 权威规则

- Session 是一等实体，UI 只持有 attachment。
- `Create / Attach / Detach / Reattach / Replay / Terminate` 必须共享同一生命周期语义。
- 渲染器、Tauri Host、Web Console 都不能成为 session 真相源。
- 控制面不得侵入 PTY 字节流热路径。

### 4.3 控制通道与热路径通道分离

- 控制动作允许走 `Tauri command`、HTTP 或 gRPC 等粗粒度通道。
- 终端热路径必须走流式通道：Windows 用 `Named Pipe`，Ubuntu/macOS 用 `Unix Domain Socket`，Web Console/Server 用 `WebSocket` 或等价流协议。
- 不允许用高频 IPC invoke 直接承载大吞吐终端字节流。

## 5. 平台适配标准

### 5.1 Windows Desktop

- 本地终端内核必须以 `ConPTY` 为权威后端。
- 必须显式处理 `cmd / PowerShell / pwsh` 的编码、环境变量、工作目录、退出码与路径差异。
- 剪贴板、窗口控制、拖拽区、快捷键、IME 走 Windows 原生适配。
- WSL 入口可以作为 profile/target，但不允许让 WSL 语义污染本地 Session 契约。

### 5.2 Ubuntu Desktop

- 本地终端内核必须以 `POSIX PTY` 为权威后端。
- 官方 Ubuntu 桌面基线锁定 `Ubuntu LTS`，其他桌面 Linux 发行版只能在明确补充兼容矩阵后再承诺。
- 必须显式处理 shell 初始化、locale、字体依赖、Wayland/X11 差异和系统包依赖。

### 5.3 macOS Desktop

- 本地终端内核必须以 `POSIX PTY` 为权威后端。
- 必须显式处理 GUI 启动时的登录 shell 环境导入、权限弹窗、快捷键、输入法与字体回退。
- 发布物必须纳入签名、公证和升级验证，不允许以开发态运行代替生产态证明。

### 5.4 Server Mode

- `server` 只负责远程 session、broker、runtime node、审计、权限、部署与运维，不接管桌面窗口职责。
- `runtime node` 以 `Ubuntu Server` 为官方主机，复用共核 `Session / Replay / Connector` 契约，但拥有独立宿主生命周期与安全模型。

## 6. 分层标准

```text
Host
  -> Shell
  -> Feature Packages
  -> Core / Infrastructure / Types / UI / i18n / Commons
  -> Rust Terminal Kernel / PTY Adapter / Connector Runtime / Storage
```

- Host：只做平台启动、窗口、权限、sidecar、桥接。
- Shell：只做 tabs/header、布局、路由、命令入口与全局状态编排。
- Feature：按 terminal、sessions、resources、settings、diagnostics 等域组织。
- Foundation：共享类型、服务、UI、配置、国际化与轻量公用组件。
- Rust Terminal Kernel：承载 PTY、Session、Replay、Recovery、Connector、CLI Host 与诊断热路径。

## 7. 当前优势

- 产品主心智已经明确收口到 terminal-first，而非 IDE/dashboard。
- `Session` 权威模型、原生 CLI 集成边界与包命名规范已冻结。
- 桌面自定义 header 与 terminal stage 基线已成立，后续可以围绕真正终端能力继续压实。

## 8. 当前关键差距

- 仍需把 `ConPTY / POSIX PTY / Stream Channel` 差异写成实现硬约束，而不是抽象口号。
- 仍需把目标三元组、签名/公证、Ubuntu LTS 基线、`Ubuntu Server` 基线写入发布门禁。
- 仍需用显式兼容矩阵证明 `Windows / Ubuntu / macOS` 与 `x64 / arm64` 的真实能力，而不是停留在“理论可支持”。

## 9. 设计红线

- 不允许宣称“四态完全共享”。
- 不允许 UI 成为 PTY、scrollback、replay 的真相源。
- 不允许 `src-tauri` 演变为巨型业务层。
- 不允许 `docker / k8s` 引入第二套 terminal 业务实现。
- 不允许把 `ConPTY` 与 `POSIX PTY` 差异淹没在单个“万能平台适配器”中。
- 不允许跨包穿透内部实现，所有消费只能经包根公共面。

## 10. 评估标准

| 评估项 | 合格线 | 领先线 | 当前要求 |
| --- | --- | --- | --- |
| 架构清晰度 | 双主态、两包装、分层清楚 | 与目录、crate、API、发布链完全映射 | `L4` |
| 平台纪律 | 明确 `ConPTY / POSIX PTY / server runtime` 边界 | 每条能力都附带平台差异与目标证据 | `L4` |
| 共核约束力 | 契约、错误、事件、发布元数据统一 | 模式切换只改变适配层与交付层 | `L4` |
| 可交付性 | 构建、测试、发布、回滚链可落地 | 支撑正式商业化交付 | `L4` |

## 11. 结论

`sdkwork-terminal` 的正确架构不是“四态一套宿主”，而是“桌面/服务端双主态 + Docker/K8s 两包装 + 共核契约 + 平台显式适配 + Rust 终端内核”的专业 terminal 体系。
