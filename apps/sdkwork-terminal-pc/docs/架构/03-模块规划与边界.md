# 03-模块规划与边界

## 1. 规划原则

- 一个模块只解决一个层次的问题。
- `Host / Shell / Feature / Foundation / Rust Runtime / Delivery` 必须分层清晰。
- `Session`、`Replay`、`Connector`、`CLI Host`、`Control Plane` 都必须有唯一真相源。
- 共核层只共享模型与契约；平台适配层与交付层必须显式分离。
- 目录命名标准与包名标准必须分离定义，避免再次混淆。

## 2. 模块总分区

### 2.1 Host 层

- `apps/desktop`
  - 桌面入口，仅负责 Tauri 窗口、权限、桌面初始化、sidecar/daemon 引导。
  - `package.json.name = @sdkwork/terminal-desktop`
- `apps/web`
  - Web Console 入口，仅负责浏览器端壳层与 server 接入。
  - `package.json.name = @sdkwork/terminal-web`
- `src-tauri/`
  - 桌面宿主组合层，只允许窗口、权限、桥接、sidecar、桌面生命周期逻辑。

### 2.2 Shell 层

## 3. 平台适配边界

### 3.1 Desktop 平台适配

- Windows
  - 使用 `ConPTY`、Windows 窗口控制、Windows 凭证与路径语义。
- Ubuntu
  - 使用 `POSIX PTY`、Ubuntu 桌面包依赖、Wayland/X11 剪贴板与 shell 初始化语义。
- macOS
  - 使用 `POSIX PTY`、macOS 签名/公证、登录 shell 环境导入、系统快捷键语义。

规则：

- 平台适配逻辑只能存在于 `src-tauri/`、`sdkwork-terminal-pty-runtime`、`sdkwork-terminal-infrastructure` 的平台边界中。
- Feature 与 UI 不允许直接感知 `ConPTY`、socket 路径、系统凭证库或安装器细节。

### 3.2 Server 平台适配

- 官方 Server 基线是 `Ubuntu Server x64 / arm64`。
- `control-plane`、`runtime-node`、`broker`、`deployments/docker`、`deployments/k8s` 只围绕 `Ubuntu Server` 主机基线建设。
- `docker / k8s` 只能包装 `server`，不能复制 terminal 业务逻辑。

## 4. 核心运行时边界

### 4.1 Terminal Core / PTY Runtime

职责：

- `terminal-core` 负责屏幕模型、文本布局、search、scrollback、selection、reflow。
- `pty-runtime` 负责 `ConPTY / POSIX PTY`、进程生命周期、resize、输入输出热路径。
- `pty-runtime` 现同时承担“显式命令 PTY bootstrap”能力：本地 shell profile 与 `ssh / docker exec / kubectl exec` 等 system-cli interactive 进程共用同一条 Rust PTY spawn substrate。

禁止：

- UI 页面直接感知 PTY 管理细节。
- `terminal-core` 直接依赖具体前端框架。
- `src-tauri` 直接承载终端字节流热路径。

### 4.2 Session Runtime / Replay Store

职责：

- `session-runtime` 负责 Session 状态机、ownership、attach/detach/reattach/recovery。
- `replay-store` 负责录制、回放、游标、索引、冷热分层。

禁止：

- UI 缓存成为 Session 真相源。
- Replay 数据模型与 Session 生命周期语义分叉。

### 4.3 Resource Connectors / AI CLI Host

职责：

- `resource-connectors` 负责 SSH / Docker / Kubernetes / Remote Runtime。
- `ai-cli-host` 负责 CLI 检测、版本、认证状态、模板、启动、恢复、诊断。

禁止：

- `ai-cli-host` 抽象 AI 产品语义。
- connector 层绕过统一协议与错误语义直接暴露实现细节。
- 将 `ssh / docker / kubectl` 的系统差异直接泄漏到 UI。

### 4.4 Control Plane / Runtime Node

职责：

- `control-plane` 负责用户、工作区、策略、审计、发布治理，以及桌面/服务端共享的会话 admission/bootstrap 真逻辑。
- `runtime-node` 负责 Server Mode 运行节点。
- `control-plane` 当前还负责 connector interactive session 共享编排：`connect preflight -> Session admission -> attachment -> PTY spawn -> launch resolution / spawn-failure replay`。

禁止：

- 控制面进入 PTY 热路径。
- Runtime Node 自建第二套 Session 模型。
- Web Console 直接定义私有协议，绕过共核 contracts/protocol。

## 5. Workspace 与命名基线

```text
apps/sdkwork-terminal
  docs/
  apps/
  packages/
  crates/
  deployments/
  tools/
```

命名硬约束：

- `packages/` 目录名必须为 `packages/sdkwork-terminal-xxx`
- `package.json.name` 必须为 `@sdkwork/terminal-xxx`
- Rust crate 名继续使用 `sdkwork-terminal-xxx`
- 导入名只能使用 `@sdkwork/terminal-xxx`

根配置冻结：

- `package.json`
- `pnpm-workspace.yaml`
- `tsconfig.base.json`
- `Cargo.toml`
- `.gitignore`

## 6. 依赖规则

- `host -> shell -> feature -> foundation`
- `feature -> contracts/core/infrastructure/ui`
- `desktop/web host -> runtime client -> local daemon or server api`
- `runtime service -> protocol/domain/storage port`

禁止：

- Feature 直接依赖其他 Feature 的内部实现。
- UI 直接依赖数据库、SSH、K8s、Docker 驱动。
- Host 直接拼装业务状态。
- Control Plane 直接依赖终端渲染实现。
- 桌面平台适配逻辑向 Web Console 反向渗透。

## 7. 组件化设计要求

- 页面组件只负责表达，不负责协议、持久化或系统能力。
- 服务组件负责编排，不负责视觉表现。
- Terminal Adapter 必须保持独立边界，避免渲染器实现泄漏到页面层。
- `src-tauri` 只做宿主桥接，不承载业务域知识。
- `apps/web` 只消费共核 contracts 和 server API，不消费桌面私有 bridge。

## 8. 评估标准

| 评估项 | 合格线 | 领先线 | 当前要求 |
| --- | --- | --- | --- |
| 模块边界清晰度 | `Host / Shell / Feature / Foundation / Runtime / Delivery` 分层成立 | 平台适配层与共核层完全分离 | `L4` |
| 命名标准一致性 | 目录、包名、导入名分离明确 | 文档、代码、测试、锁文件全一致 | `L4` |
| 依赖纪律 | 无明显跨层穿透 | 可用脚本/CI 自动校验 | `L4` |

## 9. 结论

当前模块规划的关键不是继续扩包，而是把“共核层稳定、平台层显式、交付层独立、真相源唯一”四条纪律压实到代码、测试与发布链的同一标准上。

## 2026-04-10 Supplement - Runtime-Node Step 06 Boundary

- `crates/sdkwork-terminal-runtime-node` 在 Step 06 只允许承担 server-side `SessionRuntime` bootstrap 与 recovery diagnostics，不允许提前引入 broker、调度器或第二套 Session 模型。
- 当前冻结的最小职责为：`create_runtime_node_session_runtime(...)` 返回 `SessionRuntime + RuntimeNodeRecoveryDiagnostics`，显式给出 `hostMode / platformFamily / cpuArch / runtimeLocation / storageSurface`。
- `runtime-node` 的 `session-runtime.sqlite3` 只是一条 shared recovery truth path；`docker / k8s` 只包装该 server truth path，不得复制新的持久化语义。
