# 12-安装、部署、发布与商业化交付标准

## 1. 交付目标

交付体系必须同时保证：

- 可安装
- 可启动
- 可升级
- 可回滚
- 可验证
- 可商业化交付

## 2. 交付模型

- `desktop` 与 `server` 是主交付态，但二者只共享共核契约、配置模型、观测字段、发布元数据，不共享宿主实现。
- `docker / k8s` 只包装 `server`，不复制 terminal 业务逻辑。
- 不允许以“一个安装包兼容所有平台/架构”作为目标；专业交付必须按目标三元组分别管理。

## 3. Desktop 产物矩阵

| 平台 | 目标三元组 | 标准产物 | 说明 |
| --- | --- | --- | --- |
| Windows x64 | `x86_64-pc-windows-msvc` | `MSI` / `NSIS` | 独立签名、独立 updater 元数据 |
| Windows arm64 | `aarch64-pc-windows-msvc` | `MSI` / `NSIS` | 独立签名、独立 updater 元数据 |
| macOS arm64 | `aarch64-apple-darwin` | `.app` + `.dmg` | 必须签名与公证 |
| macOS x64 | `x86_64-apple-darwin` | `.app` + `.dmg` | 必须签名与公证 |
| Ubuntu x64 | `x86_64-unknown-linux-gnu` | `.deb` + `AppImage` | 官方 Ubuntu 桌面基线 |
| Ubuntu arm64 | `aarch64-unknown-linux-gnu` | `.deb` + `AppImage` | 官方 Ubuntu 桌面基线 |

规则：

- `Ubuntu` 是唯一官方桌面 Linux 发行版基线；其他发行版只能在额外兼容矩阵完成后再扩展承诺。
- `universal` 包只能作为附加优化产物，不能替代单架构产物。
- auto-update 必须按平台、架构、通道分别下发元数据。

## 4. Server / Docker / Kubernetes 产物矩阵

| 形态 | 目标三元组 / 架构 | 标准产物 | 说明 |
| --- | --- | --- | --- |
| Server Binary | `x86_64-unknown-linux-gnu` | `tar.gz` / `deb` | 面向 Ubuntu Server/systemd |
| Server Binary | `aarch64-unknown-linux-gnu` | `tar.gz` / `deb` | 面向 Ubuntu Server/systemd |
| Docker | `linux/amd64` | OCI image | 来自 server 同一代码基 |
| Docker | `linux/arm64` | OCI image | 来自 server 同一代码基 |
| Kubernetes | `linux/amd64` + `linux/arm64` | Helm Chart + multi-arch image | 只编排 server |

## 5. 发布元数据标准

每次发布至少包含：

- `release-manifest.json`
- 版本号、提交号、构建时间、通道、目标三元组
- 支持平台/架构矩阵
- 依赖摘要、SBOM、校验和、签名/公证结果
- 升级兼容矩阵、数据迁移说明、回滚说明
- 已知风险与平台限制

## 6. 升级与回滚

- Desktop：升级前执行 preflight，升级后执行 shell/connector/replay smoke；失败必须能回滚到前一版本。
- Server：升级前执行 schema/readiness 检查，升级后执行 API/broker/runtime node smoke；失败必须能快速回滚镜像或二进制。
- Docker/K8s：回滚使用同一 `release-manifest` 与镜像标签策略，不允许维护第二套回滚逻辑。
- 数据迁移必须可追踪、可幂等、可审计。

## 7. 发布通道

- `nightly`：高频迭代验证，允许平台证据不完整，但必须显式标注。
- `beta`：扩大验证面，`P0` 矩阵必须通过。
- `stable`：`P0` 全通过，且未通过的 `P1` 目标必须有明确风险说明；若要宣称“全平台完成”，`P1` 也必须通过。

## 8. 商业化交付分层

| 版本层级 | 范围 |
| --- | --- |
| Community | Desktop 单机终端、基础 AI CLI 托管、本地连接器 |
| Team | Server Mode、自托管、基础 RBAC、审计、共享配置 |
| Enterprise | SSO、策略治理、细粒度权限、集群交付、合规增强 |

## 9. 评估标准

| 评估项 | 合格线 | 领先线 | 当前要求 |
| --- | --- | --- | --- |
| 交付完整度 | Desktop/Server/Docker/K8s 都能产生产物 | 每个目标三元组都有签名、验证与回滚证据 | `L4` |
| 发布可信度 | 元数据、SBOM、校验和齐全 | 平台级安装与升级证据长期归档 | `L4` |
| 商业化准备度 | Community/Team/Enterprise 边界明确 | 可进入正式商业交付 | `L4` |

## 10. 结论

专业 terminal 的交付不是“能打包一个 Tauri 应用”就结束，而是要把桌面、服务端、容器、集群的发布链全部建立在统一元数据和独立平台证据之上，形成真正可商业化的交付体系。
