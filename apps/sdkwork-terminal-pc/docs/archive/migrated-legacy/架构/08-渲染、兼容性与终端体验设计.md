# 08-渲染、兼容性与终端体验设计

## 1. 目标

冻结 `sdkwork-terminal` 的终端舞台标准，确保产品始终是“Windows Terminal 级 tabs/header + 真实 terminal stage”，而不是“黑色面板 + 输入框”。

## 2. 产品基线

- 首页恒定为 `顶部 custom header/tabs + 剩余区域 terminal stage`。
- `tab` 是 shell session attachment，不是业务页面导航。
- Desktop 必须隐藏系统标题栏，使用应用自定义 header。
- `+ / dropdown` 少 tab 时紧跟 tabs，空间不足或发生溢出时才 dock 到右侧。
- terminal stage 必须 full-bleed；不允许外层白边、卡片留白、dashboard 面板回归。

## 3. 真正的终端渲染分层

```text
xterm onData / focus / selection
  -> desktop runtime bridge client
  -> Tauri IPC
  -> Rust LocalShellSessionRuntime
  -> ConPTY / POSIX PTY
  -> shell echo / TUI output
  -> runtime event + replay
  -> xterm writeRaw
```

- `xterm.js` 只负责 renderer、selection、search、clipboard、viewport 测量。
- React 只持有 tab attachment 与 UI 状态，不持有 session 真相。
- `src-tauri` 只做 IPC、窗口、权限、事件桥接，不承载 PTY 热路径。
- Rust `sdkwork-terminal-pty-runtime` 必须持有 session writer、resize、terminate、reader pump。
- `sdkwork-terminal-session-runtime` 负责 replay、cursor、恢复证据，不反向替代 PTY 真相。

## 4. 输入主链硬约束

### 4.1 Desktop

- 输入必须直接来自 xterm `onData`，不能走 HTML input 回填。
- tab 首屏挂载完成后必须主动聚焦 terminal，否则用户会感知为“无法输入”。
- runtime `binding` 期间的原始按键不能丢；必须先缓存在 attachment 状态，并在 session 进入 `running` 后按顺序冲刷到 PTY。
- Enter 必须写入 `\r` 到 PTY；普通字符按字节实时写入；Backspace/方向键保留原始控制序列。
- Desktop tab 不允许退回伪 prompt；未绑定 runtime 时只允许“启动中”态，不允许假装是真终端。
- `binding` 或缓冲输入未冲刷完成时，terminal stage 必须给出终端原生状态提示，避免用户误判为“黑屏/卡死/无法输入”。
- `failed / exited` 时 terminal stage 必须给出恢复动作；至少支持同 tab `Restart shell`，不能只留一条死状态提示。

### 4.2 Web

- Web 可使用 snapshot/fallback，但语义上仍是 terminal stage。
- Web fallback 不能污染 Desktop 真终端主链实现。

## 5. 视觉与交互标准

### 5.1 Tabs / Header

- tabs 深色、低噪音、信息密度高；active tab 有顶部 accent 与底部遮罩。
- close 按钮 inactive 默认隐藏，仅 active 或 hover 可见。
- header controls、window controls、tab close 必须有 hover feedback。
- window controls 必须支持 minimize / maximize / restore / close。

### 5.2 Terminal Stage

- 字体基线：`Cascadia Code / Cascadia Mono / Consolas`。
- 终端背景、xterm viewport 背景、panel 背景必须统一，避免割裂。
- Desktop 首屏必须能点击聚焦，也必须在挂载后自动聚焦。
- search overlay 必须浮于 terminal stage 之上，不能破坏 terminal 布局。
- 启动中状态提示必须是 overlay，不得挤占 terminal stage 高度，不得引入白边或工具栏感。

### 5.3 Scrollbar

- tabs 滚动条默认隐藏。
- terminal viewport 滚动条必须细、暗、低噪音，hover 后再增强可见性。
- 滚动条样式应接近专业 terminal，而不是浏览器默认亮色滚动条。

## 6. 平台兼容基线

| 平台 | PTY 基线 | 必验能力 |
| --- | --- | --- |
| Windows | `ConPTY` | `powershell / pwsh / cmd`、焦点、输入回显、复制粘贴、窗口尺寸变化 |
| Ubuntu | `POSIX PTY` | `bash / zsh / fish`、locale、CJK、IME、resize |
| macOS | `POSIX PTY` | 登录 shell 环境、`Command` 键位、IME、clipboard |

## 7. 当前实现事实

- `sdkwork-terminal-pty-runtime` 已具备本地交互式 shell session 的创建、输入写入、resize、terminate 与 reader event pump。
- `sdkwork-terminal-shell` 已改为真实 terminal-first 页面，保留 tabs、自定义 header、window controls 与 xterm terminal stage。
- xterm viewport 已补齐 search、selection、copy/paste、scrollbar 收口与首屏主动 focus。
- Desktop 预启动阶段输入已改为“先缓存、后冲刷”，避免新 tab 启动期按键直接丢失。
- Desktop terminal stage 已新增 `binding / pending input` 原生状态提示，在空白启动窗口期给出 shell 启动与输入排队事实。
- Desktop terminal stage 已新增 `Restart shell` 恢复动作，tab 在 `failed / exited` 后可直接原位重启并重新绑定 PTY。
- `sdkwork-terminal-infrastructure` 已显式启用 xterm Unicode 11 provider，保证字符宽度能力与 TypeScript 构建一致。
- `cargo test -p sdkwork-terminal-pty-runtime --test local_shell_session_runtime` 已验证 Rust PTY session 基本交互链可用。

## 8. 剩余差距

- 仍缺 `Windows / Ubuntu / macOS` 三平台人工 smoke 证据。
- 仍需补齐 `TUI / CJK / IME / bracketed paste / OSC / alternate screen` 矩阵。
- `failed / exited` 的手动恢复已产品化，但自动恢复策略与跨平台恢复证据仍未闭环，且不能破坏 terminal-first 主舞台。

## 9. 验收标准

| 项 | 标准 |
| --- | --- |
| 页面形态 | 始终保持 `tabs/custom header + terminal stage` |
| 输入主链 | `xterm onData -> bridge -> Rust PTY writer -> shell echo` 可工作 |
| 焦点 | 首屏自动聚焦，点击后可再次稳定聚焦 |
| 渲染 | Desktop 使用真实 PTY 输出，不退回伪输入框 |
| 交互 | search / selection / copy / paste / resize / tab 切换可用 |
| 滚动条 | tabs 隐藏，terminal viewport 细化且低噪音 |
| 平台 | Windows / Ubuntu / macOS 至少具备最小可验证证据 |

## 10. 结论

## 2026-04-10 补充收口：启动失败恢复闭环

- Desktop shell 启动恢复只对 `bootstrap failure` 自动重试 `1` 次；`exited` 不自动拉起，避免偏离真实终端预期。
- `retrying` 是前端 terminal shell/runtime attachment 的显式状态语义，不替代 Rust PTY 真相层，不写回 host/runtime protocol。
- `binding / retrying` 期间的 xterm 原始输入必须统一进入 tab 级 `runtimePendingInput`；即使尚未拿到 `sessionId`，也不得丢输入。
- runtime 进入 `running` 后再顺序 flush `runtimePendingInput -> PTY write`；重试前后 flush 顺序不能乱。
- `failed` 且已消耗自动重试后，terminal overlay 必须明确提示 `Automatic retry was exhausted`，并提供同 tab `Restart shell`。
- 对 `failed` 启动失败 tab 执行 `Restart shell` 时，若存在 queued input，默认保留；对正常 `exited` tab 重启可清空 pending input。
- 本轮新增验收点：启动失败自动重试一次、`retrying` 视觉状态正确、重试中继续输入不丢失、失败后手动重启仍保留 queued input。

## 2026-04-10 补充收口：复制/粘贴语义与 bracketed paste

- 所有 clipboard paste 入口必须优先走 xterm driver 的原生 `paste(...)`，不能直接把文本注入 `onViewportInput(...)`。
- tab 级 `Copy` 不能读取陈旧 shell 快照；必须优先读取当前 xterm viewport 的实时 selection，仅在 viewport 尚未挂载时才允许回退到 tab snapshot。
- `Ctrl/Cmd + Shift + A` 不能只是把焦点重新放回 terminal；必须真正执行 viewport `selectAll()`，否则会破坏后续复制链路和专业 terminal 预期。
- Windows 终端习惯快捷键也必须成立：`Ctrl+Insert` 复制、`Shift+Insert` 粘贴，应与 `Ctrl/Cmd + Shift + C/V` 共享同一条 clipboard 语义，而不是分叉实现。
- `runtimeContentTruncated` 必须只是元数据；不能把 `"[...output truncated...]"` 之类的人工文本写回 `runtimeTerminalContent`，否则会污染 raw VT/TUI/alternate-screen 恢复链路。
- 目标不是“把文本发给 PTY”或“从状态里拿一份曾经的选择文本”，而是保留 xterm 对实时 selection、bracketed paste、粘贴事件时序与终端输入语义的原生处理能力。
- 至少以下入口必须统一：terminal 内快捷键复制/粘贴、terminal viewport 菜单复制/粘贴、tab 级菜单复制/粘贴。
- 至少以下复制/粘贴快捷键必须统一：`Ctrl/Cmd + Shift + C/V`、`Ctrl+Insert`、`Shift+Insert`。
- 至少以下选择入口必须统一：快捷键 `Ctrl/Cmd + Shift + A`、terminal viewport 菜单 `Select all`、以及任何 shell 级复制动作都必须落在同一 viewport selection 语义上。
- 截断提示若需可视化，必须走 overlay / status metadata，不得直接注入 PTY 输出流。
- 仅当 runtime driver 不具备原生 paste 能力时，才允许退回 input listener 兜底；仅当 viewport 未挂载时，tab 菜单 copy 才允许退回 shell snapshot；两类兜底都不能成为主路径。

专业 terminal 的关键不是“画一个终端样子”，而是把输入、输出、焦点、复制粘贴、搜索、scrollback、resize、replay 与平台 PTY 差异作为一条热路径统一设计。任何绕开 Rust PTY、退回 HTML 输入框、或忽略焦点/滚动条/回显质量的实现，都不能算完成。
## 2026-04-10 Supplement - Binary Input Fidelity

- Desktop terminal input is not `onData` only. `xterm.onBinary` is part of the same PTY hot path and must be treated as first-class terminal input.
- Any non-UTF-8-compatible xterm binary payload must travel through a bytes-safe path:
  `xterm.onBinary -> desktop bridge writeSessionInputBytes -> Tauri IPC -> Rust write_input_bytes -> PTY`.
- `String -> Tauri -> Rust String -> as_bytes()` is forbidden for xterm binary payloads whose byte identity must be preserved; this expands bytes above `0x7f` and breaks TUI / mouse-reporting semantics.
- `binding / retrying` buffering must preserve mixed text and binary write order. A flat string buffer is not sufficient for real terminal fidelity.
- Shell state may keep text-only preview metadata for overlay copy, but the actual flush order is defined by an ordered pending-input queue.
- Acceptance baseline for Step 05 now includes byte-fidelity evidence for mixed `text + binary + text` input and a desktop bridge command dedicated to raw input bytes.

## 2026-04-10 Supplement - OSC Title Fidelity

- `OSC 0 / OSC 2` title updates are part of real terminal behavior, not optional UI polish.
- Desktop tabs must prefer the live xterm title event, so shell/TUI title changes can surface in the header the way professional terminal products do.
- The accepted hot path is:
  `PTY output -> runtime replay -> xterm writeRaw -> xterm onTitleChange -> tab title update`.
- Blank or whitespace-only title events must be ignored; they must not erase a valid tab title.
- Runtime title is display state, not launch identity. Manual restart may restore the tab to its base launch title until the shell emits a new runtime title again.
- Step 05 evidence now includes source-level and model-level verification that viewport title changes can propagate into the Windows Terminal style tab strip without breaking terminal-first layout.

## 2026-04-10 Supplement - Terminal Fidelity Smoke Harness

- Step 05 must not rely on prose-only claims for `OSC / alternate screen / bracketed paste / mouse-reporting / CJK`.
- The repository now needs a repeatable smoke harness that can run inside a real terminal tab and collect raw terminal evidence without bypassing the PTY.
- Accepted harness shape:
  `terminal tab -> node tools/smoke/terminal-fidelity-probe.mjs -> VT sequence output + raw stdin capture -> evidence summary`.
- Harness output must be archivable. Repository smoke tooling should be able to emit both `--report-template --platform <platform> --shell <shell>` and `--review-template --platform <platform> --shell <shell>` outputs for `Windows / Ubuntu / macOS` runs, instead of leaving platform evidence as free-form notes.
- The harness must exercise:
  - `OSC 0 / 2` title mutation
  - alternate screen enter/exit
  - bracketed paste enable/disable
  - SGR mouse-reporting enable/disable
  - CJK / wide-character sample rendering
- Smoke harness is evidence tooling, not product chrome. It must not add dashboard UI or non-terminal control panels into the main application.
