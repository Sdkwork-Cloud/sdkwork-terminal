# 11-安全、测试与质量治理

## 1. 目标

安全、测试与质量治理必须同时回答三个问题：

1. 用户凭证、会话与执行环境是否安全。
2. `Windows / Ubuntu / macOS` 与 `x64 / arm64` 是否真的可用。
3. 发布、升级、回滚是否有证据链。

## 2. 风险模型

- 终端特有风险：命令误执行、环境变量泄露、回放泄密、恶意控制序列、剪贴板注入、路径跳转与链接欺骗。
- 桌面风险：过宽的 Tauri 权限、sidecar 注入、系统凭证库滥用、升级包污染。
- 服务端风险：多租户越权、broker/runtime node 冒用、审计缺失、镜像供应链污染。
- 跨平台风险：某个平台或架构缺少真实证据，却被误标为“已支持”。

## 3. 安全标准

### 3.1 Desktop

- Tauri capability 默认最小化，按窗口控制、shell、文件、剪贴板逐项授权。
- 系统凭证必须优先写入平台凭证库：Windows `Credential Manager`、macOS `Keychain`、Ubuntu `Secret Service/libsecret`；只有在明确不可用时才允许受控回退。
- 环境变量注入、工作目录切换、命令模板必须白名单化并可审计。
- 需要明确 `OSC 52`、文件链接、终端超链接等高风险功能的默认策略。

### 3.2 Server Mode

- 统一 `OIDC / SSO + RBAC + Audit Log`。
- `Control Plane`、`Broker`、`Runtime Node` 必须双向认证并支持证书轮换。
- API Token、Session Token、节点凭证必须分离。
- 远程 session、replay、diagnostics 导出必须具备权限、脱敏与审计链。

### 3.3 Supply Chain

- 依赖锁定、SBOM、校验和、签名、公证、镜像扫描、Helm 模板扫描缺一不可。
- 不允许以“本地能跑”替代“发布物可信”。

## 4. 测试体系

### 4.1 测试层次

- 单元测试：parser、screen model、状态机、配置优先级、错误体。
- 契约测试：IPC、HTTP、stream、connector、CLI host、release manifest。
- 集成测试：本地 shell、SSH、Docker、Kubernetes、AI CLI、replay/recovery。
- 兼容性测试：shell、TUI、CJK、IME、clipboard、resize、search、replay。
- 非功能测试：性能、压力、弱网、崩溃恢复、升级/回滚、安装包 smoke。

### 4.2 平台矩阵

| 形态 | 平台 | 架构 | 级别 | 说明 |
| --- | --- | --- | --- | --- |
| Desktop | Windows | `x64` | P0 | 首发阻塞矩阵 |
| Desktop | Ubuntu | `x64` | P0 | 首发阻塞矩阵 |
| Desktop | macOS | `arm64` | P0 | 首发阻塞矩阵 |
| Desktop | Windows | `arm64` | P1 | 扩展阻塞矩阵 |
| Desktop | Ubuntu | `arm64` | P1 | 扩展阻塞矩阵 |
| Desktop | macOS | `x64` | P1 | 扩展阻塞矩阵 |
| Server | Ubuntu Server | `x64` | P0 | 首发阻塞矩阵 |
| Server | Ubuntu Server | `arm64` | P0 | 首发阻塞矩阵 |

规则：

- `P0` 未通过，不得进入正式发布。
- `P1` 未通过，不得宣称“全平台全架构完成”。
- 没有目标证据的组合只能标记为“未验证”，不能标记为“支持”。

## 5. 发布门禁

正式发布前必须同时通过：

- 平台构建门禁：所有目标三元组构建成功。
- 终端体验门禁：shell/TUI/CJK/IME/clipboard/search/replay 基线通过。
- 安全门禁：凭证、权限、审计、脱敏、SBOM、签名/公证通过。
- 交付门禁：安装、启动、升级、回滚、镜像与 Helm smoke 通过。
- 文档门禁：`README.md`、`docs/架构/`、`docs/step/`、`docs/release/` 事实一致。

## 6. 证据链标准

- 每个目标平台都必须至少有：构建结果、启动 smoke、核心交互 smoke、失败截图或日志摘要、发布物清单。
- 每个 Step 都必须明确“本步新增了哪些平台证据，仍缺哪些平台证据”。
- release 候选必须能回答“支持哪些平台/架构、证据在哪、已知风险是什么”。

## 7. 评估标准

| 评估项 | 合格线 | 领先线 | 当前要求 |
| --- | --- | --- | --- |
| 安全设计 | Desktop/Server/Supply Chain 有完整标准 | 终端特有风险也被制度化治理 | `L4` |
| 测试完整度 | 单元/契约/集成/兼容/发布 smoke 完整 | 三平台双架构长期自动执行 | `L4` |
| 发布可信度 | 有签名、公证、SBOM、回滚链 | 所有平台证据可追溯归档 | `L4` |

## 8. 结论

`sdkwork-terminal` 的质量标准必须是“真实跨平台证据驱动”，而不是“代码结构看起来支持跨平台”。没有目标证据，就没有平台支持声明。
