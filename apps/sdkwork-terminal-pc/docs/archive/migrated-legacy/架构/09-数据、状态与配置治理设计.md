# 09-数据、状态与配置治理设计

## 1. 目标

定义本地与服务端的数据分层、状态真相来源、配置治理方式与恢复路径，确保产品长期满足高内聚、低耦合、易恢复、易审计要求。

## 2. 状态分层

### 2.1 用户可编辑配置

- 形态：`config.toml`
- 用途：Profile、工作区默认项、主题、键位、策略开关
- 原则：可版本化、可审计、可迁移

### 2.2 运行时状态

- 形态：`SQLite`
- 用途：Session 索引、Replay 片段索引、恢复位点、退出元数据
- 原则：由 runtime 写入，不由 UI 直接维护

### 2.3 服务端持久化

- `PostgreSQL`：控制面主数据
- `Redis`：热状态与协调缓存
- `Object Storage`：大体积记录、诊断包、归档对象

### 2.4 共核与非共核边界

- 共核共享的是 `Session / Replay / ExecutionTarget / Launch Template / error / observability field` 模型。
- Desktop 与 Server 不共享存储引擎实现，只共享模型、字段、迁移语义与证据口径。
- `docker / k8s` 不引入第二套持久化语义，只包装 Server 持久化链。

## 3. 当前事实

- `Step 06` 已在本地侧落地 SQLite 主链：
  - `session_index`
  - `replay_entries`
- `session_index` 已保存：
  - `sessionId`
  - `workspaceId`
  - `target`
  - `state`
  - `createdAt`
  - `lastActiveAt`
  - `modeTags`
  - `tags`
  - `lastAckSequence`
  - `exitCode`
- `replay_entries` 已保存：
  - `sessionId`
  - `sequence`
  - `kind`
  - `payload`
  - `occurredAt`
- 当前持久化主链已能表达本地 Session/replay 真相，但不等同于“Desktop/Server 共用同一存储实现”。

## 4. 状态真相规则

- Session 真相只允许来自 `session-runtime`
- React UI 只持有 derived view model
- Attachment 状态不能反向覆盖 Session 真相
- Release、审计、回滚只认 runtime 持久化与明确验证结果
- 持久化记录必须显式保留 `platformFamily`、`cpuArch`、`hostMode`、`targetType` 或等价字段，禁止用模糊“跨平台”标识替代

## 5. 恢复规则

- 如果 Session 仍存活，则优先 `Reattach`
- 如果只剩历史数据，则走 `Replay`
- `lastAckSequence` 是恢复的重要边界，不允许前端私自推断

## 6. 后续扩展边界

- `config.toml` 的 schema 治理、热更新、迁移工具后续继续增强
- 服务端状态治理在 `Step 10-11` 深化
- 诊断包、归档与跨节点恢复在 `Step 11-13` 收口
- Desktop 继续以 `config.toml + SQLite` 为本地基线；Server 继续以 `PostgreSQL + Redis + Object Storage` 为服务端基线

## 7. 评估结论

| 项目 | 标准 | 当前结论 |
| --- | --- | --- |
| 本地运行时状态 | SQLite 索引与恢复位点成立 | `L4` |
| UI / Runtime 状态分离 | UI 不再持有会话真相 | `L4` |
| Desktop / Server 状态边界 | 共核共享模型，不共享存储实现 | `L4` |
| 配置治理 | `config.toml + SQLite` 分层已冻结 | `L3` |
